## llama-server

`llama-server` 是 `llama.cpp` 项目中的核心组件，用于将大语言模型部署为服务。你可以把它看作一个**轻量级、高性能的本地大模型服务框架**，它把复杂的模型推理过程封装成了简单的 HTTP 接口，让你可以通过 API 或 Web 界面与模型交互。

### 原理介绍与基础架构

`llama-server` 的架构设计主要围绕“高性能”和“轻量级”展开，其核心构成如下：

1.  **底层通信**：它使用一个纯 C++ 编写的 `cpp-httplib` 库来处理网络请求，这使得通信过程非常轻量且高效。
2.  **核心处理**：服务在内部划分为两个主要逻辑部分：`server_http_context` 负责处理来自客户端的 HTTP 请求和响应；`server_context` 则负责实际的模型推理任务管理。
3.  **流式响应**：为了实现极快的交互体验，`llama-server` 支持**流式传输 (Streaming)**。推理生成的文本（Token）会实时地、逐字地推送给客户端，而不是等到全部生成完再一次性返回。
4.  **多模型管理（路由模式）**：新版本引入了重要的“路由模式 (Router Mode)”。启动时不指定具体模型，服务就能变成一个模型管理器。它能**动态加载/卸载**模型，支持**按需加载**和基于 **LRU（最近最少使用）** 的缓存淘汰策略，实现了多模型之间的毫秒级切换，且模型运行在独立的进程中，实现了进程隔离。

### 使用方法

`llama-server` 的使用分为**单模型模式**和**路由模式（多模型）**，以及通过 API 进行交互。

#### 1. 基础启动与配置

*   **单模型模式**：直接在启动命令中指定模型，服务会与这个模型绑定。

    ```bash
    ./llama-server \
        -m /path/to/your/model.gguf \   # 指定模型文件路径
        --host 0.0.0.0 \                # 监听所有网络接口
        --port 8080                     # 指定服务端口
    ```
    

*   **路由模式（多模型）**：不指定具体模型，服务会扫描指定目录或缓存中的模型，按需加载。

    ```bash
    ./llama-server \
        --models-dir /path/to/models   # 指定存放GGUF模型的目录
    ```
    

#### 2. API 调用与接口

服务启动后，会提供兼容 OpenAI 格式的 API 接口，方便集成。你可以使用 `curl` 命令测试服务是否正常。

**发送聊天请求**：
```bash
curl -X POST http://localhost:8080/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
        "model": "your-model-name",
        "messages": [{"role": "user", "content": "Hello!"}],
        "max_tokens": 150,
        "temperature": 0.7
    }'
```


**管理模型（路由模式下）**：
你还可以通过额外的 HTTP 接口来手动管理模型的生命周期。

*   **查看所有可用模型**：`curl http://localhost:8080/models`
*   **手动加载模型**：`curl -X POST http://localhost:8080/models/load -d '{"model": "model-name.gguf"}'`
*   **手动卸载模型**：`curl -X POST http://localhost:8080/models/unload -d '{"model": "model-name.gguf"}'`

### 如何选择优化模型启动参数

优化参数的目的是为了在硬件限制下达到最佳性能。选择时需要根据你的**硬件配置**（尤其是显存和CPU核心数）和**使用场景**来决定。

我来为你详细汇总 `llama-server` 每个参数的作用原理和使用场景。这份完整指南将按功能模块分类，方便你查阅。

## 📚 一、通用基础参数

### 1.1 帮助与版本
| 参数                | 作用                 | 原理                             | 使用场景                          |
| :------------------ | :------------------- | :------------------------------- | :-------------------------------- |
| `-h, --help`        | 显示帮助信息         | 打印所有可用参数列表             | 初次使用或忘记参数时查阅          |
| `--version`         | 显示版本和构建信息   | 输出编译时的版本号和CUDA/CPU特性 | 确认llama.cpp版本，排查兼容性问题 |
| `--completion-bash` | 生成Bash自动补全脚本 | 输出可source的补全脚本           | 方便命令行操作，提高效率          |

### 1.2 CPU与线程控制
| 参数                     | 作用            | 原理                                     | 使用场景                                      |
| :----------------------- | :-------------- | :--------------------------------------- | :-------------------------------------------- |
| `-t, --threads N`        | CPU推理线程数   | 控制生成阶段使用的CPU核心数              | CPU推理时优化速度；GPU推理时可设小些（如4-8） |
| `-tb, --threads-batch N` | 批处理线程数    | 控制prompt处理阶段的线程数               | 可单独优化首token延迟，通常与`-t`相同         |
| `-C, --cpu-mask M`       | CPU亲和性掩码   | 用十六进制指定允许使用的CPU核心          | 在多核系统上隔离核心，减少上下文切换          |
| `-Cr, --cpu-range lo-hi` | CPU范围亲和性   | 用范围指定允许使用的CPU核心              | 更直观地指定核心范围，如`0-7`                 |
| `--cpu-strict`           | 严格CPU放置     | 强制进程只运行在指定核心上               | 需要极致性能隔离时使用                        |
| `--prio N`               | 进程/线程优先级 | 设置调度优先级：-1低,0正常,1中,2高,3实时 | 保证服务响应速度，避免被其他进程抢占          |
| `--poll N`               | 轮询等待级别    | 0-100，控制工作等待时的CPU轮询程度       | 高值减少延迟但增加CPU占用，默认50             |

> **使用场景示例**：
> ```bash
> # 16核系统，用0-7核做推理，8-15核做批处理
> -t 8 -tb 8 -Cr 0-7 -Crb 8-15
> ```

## 🧠 二、模型与上下文参数

### 2.1 模型加载
| 参数                        | 作用             | 原理                 | 使用场景                             |
| :-------------------------- | :--------------- | :------------------- | :----------------------------------- |
| `-m, --model FNAME`         | 指定模型文件路径 | 加载GGUF格式模型     | 单模型部署时必填                     |
| `-mu, --model-url URL`      | 模型下载URL      | 从URL下载模型到缓存  | 自动获取模型，无需手动下载           |
| `-hf, --hf-repo`            | Hugging Face仓库 | 自动从HF下载GGUF模型 | 方便从HF获取模型，支持自动下载mmproj |
| `-hff, --hf-file`           | HF仓库中的文件名 | 指定要下载的具体文件 | 当仓库有多个文件时精确选择           |
| `-hf-token`                 | HF访问令牌       | 用于私有模型仓库认证 | 下载需要登录的模型                   |
| `--lora FNAME`              | LoRA适配器       | 加载LoRA微调权重     | 在基座模型上应用特定任务适配器       |
| `--lora-scaled FNAME:SCALE` | 带缩放因子的LoRA | 可控制LoRA的影响强度 | 动态调整LoRA效果                     |

### 2.2 上下文与生成控制
| 参数                   | 作用              | 原理                                       | 使用场景                                  |
| :--------------------- | :---------------- | :----------------------------------------- | :---------------------------------------- |
| `-c, --ctx-size N`     | 上下文大小        | KV缓存的最大token数，决定"记忆"长度        | 长文档处理设大值（如32768），短对话设小值 |
| `-n, --predict N`      | 最大生成长度      | 单次推理最多生成的token数                  | 控制回答长度，-1为无限                    |
| `-b, --batch-size N`   | 逻辑批处理大小    | 单次处理的最大token批次数                  | 默认2048，大模型可适当增大以提高吞吐      |
| `-ub, --ubatch-size N` | 物理批处理大小    | 实际执行的最小批处理单元                   | 默认512，影响显存峰值占用                 |
| `--keep N`             | 保留上下文token数 | 保留前N个token不移出上下文                 | 保持系统提示词始终在上下文中              |
| `--rope-scaling`       | RoPE频率缩放方法  | 扩展上下文长度的数学方法：none/linear/yarn | 当需要超长上下文时使用                    |
| `--rope-scale N`       | RoPE缩放因子      | 上下文扩展倍数                             | 配合`--rope-scaling`使用，如2x扩展        |
| `--yarn-orig-ctx N`    | YaRN原始上下文    | 模型训练时的原始上下文大小                 | YaRN缩放时需指定                          |

> **使用场景示例**：
> ```bash
> # 支持128K上下文
> -c 131072 --rope-scaling yarn --rope-scale 4
> ```

## 🚀 三、GPU与性能优化参数

### 3.1 GPU卸载与层管理
| 参数                    | 作用           | 原理                                                 | 使用场景                         |
| :---------------------- | :------------- | :--------------------------------------------------- | :------------------------------- |
| `-ngl, --gpu-layers N`  | GPU层数        | 将N层Transformer放到GPU显存。可用数字、`auto`或`all` | 核心性能参数，显存允许时尽量设大 |
| `-dev, --device`        | 指定设备       | 逗号分隔的设备列表，如`cuda:0,cuda:1`                | 多GPU时指定使用哪些设备          |
| `--list-devices`        | 列出可用设备   | 打印所有可用的GPU/设备                               | 查看系统中有哪些设备可用         |
| `-sm, --split-mode`     | 多GPU拆分模式  | `none`/`layer`/`row`/`tensor`，决定如何分配模型      | 多GPU并行推理时选择最优分配策略  |
| `-ts, --tensor-split`   | 张量拆分比例   | 如`3,1`表示GPU0:GPU1=3:1分配                         | 多GPU时按显存比例分配负载        |
| `-mg, --main-gpu INDEX` | 主GPU索引      | 指定主GPU处理中间结果和KV缓存                        | `split-mode=row`时需指定         |
| `--no-host`             | 禁用主机缓冲区 | 允许使用额外缓冲区，减少主机内存占用                 | 在显存充裕时追求极致性能         |

### 3.2 显存优化
| 参数                   | 作用              | 原理                                  | 使用场景                         |
| :--------------------- | :---------------- | :------------------------------------ | :------------------------------- |
| `-ctk, --cache-type-k` | KV缓存K的数据类型 | f32/f16/bf16/q8_0/q4_0等，量化KV缓存  | 显存紧张时设为q4_0可大幅节约显存 |
| `-ctv, --cache-type-v` | KV缓存V的数据类型 | 同上，可独立设置                      | 通常与K设为相同类型              |
| `--flash-attn`         | Flash Attention   | 使用Flash Attention算法优化注意力计算 | 强烈推荐开启，显著加速长上下文   |
| `--kv-offload`         | KV缓存卸载        | 将KV缓存放到GPU或CPU                  | 默认启用；显存不足时可禁用       |
| `--repack`             | 权重重打包        | 优化内存布局减少碎片                  | 默认启用，有助于多GPU场景        |
| `--mlock`              | 内存锁定          | 将模型锁定在物理内存，防止swap        | 提高推理稳定性，避免卡顿         |
| `--mmap`               | 内存映射          | 使用内存映射加载模型，快速启动        | 默认启用；禁用可减少虚拟内存占用 |
| `--numa`               | NUMA优化          | `distribute`/`isolate`/`numactl`      | 在多CPU插槽系统上优化内存访问    |

### 3.3 自动适配（重要新特性）
| 参数               | 作用         | 原理                               | 使用场景                   |
| :----------------- | :----------- | :--------------------------------- | :------------------------- |
| `--fit`            | 自动适配显存 | 自动调整`-c`和`-ngl`以适应显存限制 | 防止OOM，推荐开启          |
| `--fit-target MiB` | 适配目标余量 | 为设备保留的显存余量（MB）         | 默认1024MB，保证系统稳定性 |
| `--fit-ctx N`      | 最小上下文   | `--fit`可设置的最小`-c`值          | 防止上下文被压缩得过小     |

### 3.4 MoE模型专用
| 参数            | 作用          | 原理                       | 使用场景                                 |
| :-------------- | :------------ | :------------------------- | :--------------------------------------- |
| `--cpu-moe`     | 全部MoE在CPU  | 将所有MoE专家权重保留在CPU | 显存严重不足时                           |
| `--n-cpu-moe N` | 前N层MoE在CPU | 将前N层的MoE专家放在CPU    | 平衡显存与速度，你的Qwen 35B模型可能需要 |

## 🎯 四、采样与控制参数

### 4.1 采样器顺序
| 参数            | 作用           | 原理                                      | 使用场景                   |
| :-------------- | :------------- | :---------------------------------------- | :------------------------- |
| `--samplers`    | 采样器顺序     | 采样器执行顺序：`penalties;dry;top_k;...` | 自定义采样流程，默认已优化 |
| `--sampler-seq` | 采样器简写序列 | 如`edskypmxt`，每个字母代表一个采样器     | 快速调整采样器组合         |
| `-s, --seed`    | 随机种子       | 设置RNG种子，-1为随机                     | 需要可复现结果时固定种子   |

### 4.2 主要采样参数
| 参数                   | 作用           | 原理                              | 使用场景                            |
| :--------------------- | :------------- | :-------------------------------- | :---------------------------------- |
| `-temp, --temperature` | 温度           | 控制概率分布平滑度，越高越随机    | 创意写作用0.8-1.0，代码用0.2-0.5    |
| `--top-k`              | Top-K采样      | 只从概率最高的K个token中采样      | 默认40，值越小输出越集中            |
| `--top-p`              | 核采样         | 从累积概率达到p的最小token集采样  | 默认0.95，1.0为禁用                 |
| `--min-p`              | 最小概率采样   | 只考虑概率不低于p*max_prob的token | 默认0.05，可过滤低概率噪声          |
| `--typical-p`          | 局部典型采样   | 选择信息熵接近预期的token         | 可产生更"自然"的文本                |
| `--xtc-probability`    | XTC概率        | 以该概率剔除极端token             | 0.0禁用，用于减少重复               |
| `--mirostat`           | Mirostat采样   | 动态控制输出熵                    | 0禁用，1为Mirostat，2为Mirostat 2.0 |
| `--mirostat-lr`        | Mirostat学习率 | 控制熵调整速度                    | 默认0.10                            |
| `--mirostat-ent`       | Mirostat目标熵 | 目标信息熵值                      | 默认5.00                            |

### 4.3 惩罚参数
| 参数                  | 作用        | 原理                        | 使用场景                    |
| :-------------------- | :---------- | :-------------------------- | :-------------------------- |
| `--repeat-penalty`    | 重复惩罚    | 惩罚已出现token的重复       | 1.0禁用，1.1-1.2可减少重复  |
| `--presence-penalty`  | 存在惩罚    | 对出现过的token施加固定惩罚 | 鼓励引入新主题              |
| `--frequency-penalty` | 频率惩罚    | 对出现过的token施加累积惩罚 | 抑制高频词重复              |
| `--repeat-last-n`     | 惩罚窗口    | 只考虑最后N个token进行惩罚  | 默认64，越短越关注近期      |
| `-l, --logit-bias`    | Logit偏置   | 直接调整特定token的logit值  | 强制/禁止特定词出现         |
| `--dry-multiplier`    | DRY采样倍率 | 调节DRY采样强度             | 可有效减少重复，默认0.0禁用 |

### 4.4 语法约束
| 参数                 | 作用            | 原理                       | 使用场景             |
| :------------------- | :-------------- | :------------------------- | :------------------- |
| `--grammar`          | BNF语法约束     | 用BNF文法限制输出格式      | 生成JSON、结构化数据 |
| `--grammar-file`     | 语法文件        | 从文件读取BNF语法          | 复杂语法复用         |
| `-j, --json-schema`  | JSON Schema     | 用JSON Schema约束输出      | 需要严格JSON输出时   |
| `--json-schema-file` | JSON Schema文件 | 从文件读取JSON Schema      | 复杂Schema复用       |
| `--backend-sampling` | 后端采样        | 在设备端执行采样（实验性） | 减少CPU-GPU数据传输  |

## 🤖 六、高级推理参数

### 6.1 并行与批处理
| 参数                | 作用       | 原理                              | 使用场景               |
| :------------------ | :--------- | :-------------------------------- | :--------------------- |
| `-np, --parallel N` | 并行槽位数 | 同时处理的请求数，-1为自动        | 高并发服务需调大       |
| `--cont-batching`   | 连续批处理 | 动态将不同请求合并批处理          | 默认启用，提高吞吐量   |
| `--cache-prompt`    | Prompt缓存 | 缓存已处理的prompt                | 相同前缀复用，提高效率 |
| `--cache-reuse N`   | 缓存复用   | 通过KV移位复用缓存的最小chunk大小 | 需要prompt缓存启用     |

### 6.2 推测解码（加速推理）
| 参数                 | 作用            | 原理                                           | 使用场景                        |
| :------------------- | :-------------- | :--------------------------------------------- | :------------------------------ |
| `--spec-type`        | 推测解码类型    | `none`/`draft-simple`/`draft-eagle3`/`ngram-*` | 用小模型/ngram加速大模型推理    |
| `--model-draft`      | 草稿模型        | 用于推测解码的较小模型                         | 需要显著加速时使用              |

### 深入解读：选择哪一个？

**1. 追求极致控制与性能 → 选 llama-server**

`llama-server`直接与底层推理库`llama.cpp`交互，没有中间层，非常轻量。这使得它能提供最高的运行效率，是性能爱好者和研究人员的不二之选。你可以精细地调整每一个可能影响性能的参数，如`-ngl` (GPU加载层数)、`-t` (线程数)、`--cache-type-k` (KV缓存数据类型)等。近期引入的**路由模式**，更使其具备多模型动态加载、进程隔离的能力，向一个成熟的轻量级推理服务框架演进。

**2. 快速开发、开箱即用 → 选 Ollama**

Ollama的目标是让运行大模型变得像`docker run`一样简单。它封装了`llama.cpp`等后端引擎，为你提供了：
*   **极简的安装与使用**：一键安装，几条命令就能下载并运行模型。
*   **自动化的资源管理**：无需操心模型文件放哪，它会帮你管理下载、存储和版本。
*   **友好的开发者生态**：提供开箱即用的REST API、Python和JavaScript库，能快速集成到你的应用中。正如一些测评所示，它在易用性方面的体验远超底层工具。

### 一句话总结

*   **`llama-server`** 适合“**工程师**”，它把性能和灵活性摆在首位，让你能按需精雕细琢。
*   **Ollama** 适合“**开发者**”，它把效率和易用性放在首位，让你能快速启动并专注于应用构建。
