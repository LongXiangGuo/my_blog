RFCOMM：这是经典蓝牙特有的传输层协议。它在 L2CAP 之上，最多可以模拟 60 个同时存在的虚拟串口通道。
SPP：它是基于 RFCOMM 的应用层规范。它定义了如何建立仿真串口连接，以及如何配置波特率、数据位、停止位等（虽然蓝牙传输速率由射频决定，但波特率参数会透传给底层的物理 UART）。

>iOS 默认不支持通用 SPP：苹果的 iPhone/iPad 系统直接从底层屏蔽了标准的蓝牙 SPP 协议。如果你的硬件使用标准的 SPP，iOS 的蓝牙设置和应用是完全搜不到、也连不上的。

## BLE串口通信

在蓝牙低功耗（BLE 4.0及以上）技术中，官方协议栈中并没有像经典蓝牙那样自带现成的 SPP（标准串口协议）。
在 BLE 中实现串口通信，行业内的标准做法是：利用 BLE 的 GATT（通用属性配置文件） 架构，自定义一个包含“发送（TX）”和“接收（RX）”功能的专属服务（Service）与特征值（Characteristic），从而在应用层人工模拟出一条双向的“虚拟串口”透传通道。

**一、 BLE 串口通信的核心架构设计**
在 BLE 串口中，设备通常分为两个角色：
外设/从机（Peripheral/Server，如蓝牙透传模块、心率手环）：提供 GATT 服务，里面塞满了各种特征值（类似寄存器），等待别人来读写。
中心设备/主机（Central/Client，如智能手机、iPad）：主动扫描、连接外设，并对其中的特征值进行读写或订阅。
为了模拟串口，外设必须在内部创建一个自定义的 Service，并在其中至少包含以下两个 Characteristics（特征值）：


```
★ 蓝牙透传服务 (Custom UART Service) - 自定义 UUID
  │
  ├── 1. 接收特征值 (RX Characteristic) - [手机 -> 硬件]
  │      ├─ 属性 (Properties): WRITE 或 WRITE_WITHOUT_RESPONSE
  │      └─ 作用: 手机往这个特征值里写数据，单片机就能收到。
  │
  └── 2. 发送特征值 (TX Characteristic) - [硬件 -> 手机]
         ├─ 属性 (Properties): NOTIFY (通知) 或 INDICATE
         ├─ 描述符 (CCCD): 必须配置，允许手机开启订阅
         └─ 作用: 单片机有数据要发时，触发 Notify，手机端自动接收推送。
```

**二、 BLE 4.0 串口通信的四个关键痛点**
在用 BLE 4.0 代替经典蓝牙 SPP 做串口时，开发者必须面对以下几个底层物理限制：
1. 20字节的单包限制（ATT MTU 限制）
痛点：在传统的 BLE 4.0 规范中，默认的有效载荷（Payload）单包最大只有 20个字节。如果你一次性发送 50 个字节，超出部分会被直接丢弃。
解法：
软件分包：应用层代码必须编写分包算法，把一长串字符串拆成每 20 字节一包，在发送端循环发送；接收端再拼接。
MTU 交换：连接成功后，手机端主动发起 requestMtu(512) 指令。如果硬件模块支持，可以将单包上限扩充到 244 字节（通常需要 BLE 4.2 及以上硬件支持）。
2. 通信速率较低
痛点：BLE 4.0 的物理层速率只有 1Mbps，扣除协议开销和连接间隔延迟，实际应用层透传速率通常只有 几KB/s 到十几KB/s。
建议：绝不适合用来传输大图片、音频或大量历史 log，它只适合传输控制指令、传感器实时小数据。

3. 异步通知机制（Notify）
经典蓝牙是“数据流（Stream）”，而 BLE 是“报文（Packet）”。
当硬件（从机）想发数据给手机时，它不能直接写，而是通过 Notify 机制异步推送。手机端必须提前通过代码开启该特征值的通知监听（写入 CCCD 描述符为 0x0001），否则手机根本收不到硬件发来的数据。
4. 完美兼容 iOS（最大优势）
BLE 最大的优势在于苹果 iOS 系统完全开放。你开发的 iPhone App 不需要经过任何昂贵的 MFi 芯片认证，就可以直接连接任何一个支持 BLE 的硬件模块进行串口数据读写。
