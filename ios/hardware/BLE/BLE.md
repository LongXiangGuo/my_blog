# BLE

蓝牙技术联盟（SIG）将蓝牙技术的发展主要划分为两个泾渭分明的技术路线：经典蓝牙（BR/EDR）与低功耗蓝牙（BLE）。

**蓝牙版本演进与核心特性对比表**

| 版本 | 核心特性 | 传输速率 (理论) | 解决的核心问题与历史意义 |
| :--- | :--- | :--- | :--- |
| **v1.0 / 1.2** | 基础速率（BR: Basic Rate） | 1 Mbps | 统一无线个人局域网（WPAN）标准，实现点对点语音和数据传输，替代串口线。 |
| **v2.0 / 2.1 + EDR** | 增强速率（EDR: Enhanced Data Rate） | 3 Mbps | v2.0 引入 EDR（采用波形调制技术），大幅提升速率。v2.1 引入 SSP（安全简易配对），利用安全机制防止配对被窃听。 |
| **v3.0 + HS** | 高速传输（HS: High Speed） | 24 Mbps | 借道机制（AMP）。蓝牙负责建立连接，大数据传输时调用片上 Wi-Fi 硬件跑全速，传完再切回蓝牙。 |
| **v4.0 / 4.1 / 4.2** | 低功耗蓝牙（BLE / Smart） | 1 Mbps | 分水岭版本。引入全新不兼容的 BLE 架构，专为纽扣电池供电的 IoT 节点设计。v4.2 大幅扩展了数据包长度（DLE），为物联网爆发奠定基础。 |
| **v5.0** | 2倍速率 / 4倍距离 / 8倍广播容量 | 2 Mbps (LE 2M) | 允许在“高速率”与“远距离（LE Coded，理论超 100 米）”间权衡，广播包从 31 字节扩展到 255 字节。 |
| **v5.1 / 5.2 / 5.3** | 寻向定位 / LE Audio | 2 Mbps | v5.1 引入 AoA/AoD 测向，实现度级角度定位。v5.2 引入 LC3 编码与 LE Audio，支持低功耗真无线耳机（TWS）和广播音频（Auracast）。 |
| **v5.4** | 带响应的周期性广播 (PAwR) | 2 Mbps | 允许电子货架标签（ESL）这类数万个节点的网络与网关进行双向加密周期性通信。 |
| **v6.0 (最新)** | 信道探测 (Channel Sounding) | 2 Mbps | 引入厘米级（Centimeter-level）的高精度距离测量，彻底改变无钥匙进入（数字钥匙）的安全性和精准度，防止中继攻击。 |

## AMP

AMP 代表 Alternate MAC/PHY（替代 MAC/PHY 层），是蓝牙官方在 2009 年发布的 蓝牙 3.0 + HS (High Speed) 规范中引入的核心机制 [v3.0 + HS]。
大白话来说，它的核心思想就是：“用蓝牙的低功耗来谈恋爱（建立连接），用 Wi-Fi 的大身板来干苦力（搬运大文件）。”

当时蓝牙的理论极限速率只有 3 Mbps（2.1 + EDR 时代）[v2.0 / 2.1 + EDR]，面对早期的无线照片传输、视频同步等大数据场景显得力不从心。于是，蓝牙官方设计了 AMP 借道机制。以下是它的底层实现原理和工作生命周期：

在硬件层面，支持蓝牙 3.0 + HS 的设备内部通常有一颗组合芯片（Combo Chip），它同时集成了蓝牙控制器（Bluetooth Controller）和 Wi-Fi 802.11 控制器。
在软件协议栈层面，蓝牙官方在传统的蓝牙 L2CAP（逻辑链路控制与适配协议）层之下，插入了一个名为 PAL (Protocol Adaptation Layer，协议适配层) 的全新模块：

```
       +---------------------------------------------+

       |         上层应用层 / 业务数据流 (Profile)    |
       +---------------------------------------------+
                              │
                              ▼
       +---------------------------------------------+

       |         蓝牙核心 L2CAP 层 (数据分流闸)        |
       +---------------------------------------------+
                              │
               ┌──────────────┴──────────────┐
               │ (小数据/控制流)              │ (大数据流 Handover)
               ▼                             ▼
       +---------------+             +---------------+

       | 传统蓝牙基带   |             |   AMP - PAL   | (协议适配层)
       | (BR / EDR)    |             +---------------+
       +---------------+                     │
               │                             ▼
               │                     +---------------+
               │                     | 802.11 Wi-Fi  | (借用 Wi-Fi 物理层)
               ▼                     +---------------+
       [ 传统蓝牙无线电波 ]                 [ 5GHz / 2.4GHz 无线电波 ]
```

当两个设备（例如手机和车机、或两台手机）准备传输一个 100MB 的大文件时，系统会全自动执行以下四个阶段：
**阶段 1：蓝牙慢速建连与审查**
动作：两个设备首先通过传统的经典蓝牙进行标准的 SSP 安全简易配对 [v2.0 / 2.1 + EDR]。
查询：连接建立后，手机通过蓝牙控制通道询问对方：“你支持 AMP 借道机制吗？你本地有哪些可用的高速通道（AMP Managers）？”
结果：如果双方都报告自己内置了支持 802.11 的 AMP 控制器，则握手成功，随时准备切流。
**阶段 2：动态握手流重定向 (Channel Move / Handover)**
触发：当用户点击“发送大文件”时，上层应用向 L2CAP 层灌入海量字节。L2CAP 发现数据量超过了经典蓝牙 EDR 链路的承载极限（队列积压）。
对暗号：蓝牙核心向 PAL 层 发出指令。此时，双方的蓝牙控制器在后台通过蓝牙链路进行协商，生成一组临时的、专用于这次传输的 Wi-Fi 物理层连接参数（包括信道、加密秘钥等）。
点火：参数对齐后，蓝牙物理层发出指令，直接强行唤醒（Power On）两端原本处于关闭/休眠状态的 Wi-Fi 芯片硬件。
**3. 阶段 3：Wi-Fi 物理层全速对轰（不走 TCP/IP）**
核心误区澄清：这里的借用 Wi-Fi，绝对不是让你的手机去连一个常规的无线路由器（不走标准的 TCP/IP、没有 DHCP 自动分配 IP 这一套）。
原理：两端的 Wi-Fi 芯片被唤醒后，直接切入 802.11 PAL 专用点对点模式（类似于 Wi-Fi Direct 的底层变种）。
传输：上层的蓝牙数据包（L2CAP 包）直接被包裹进 Wi-Fi 的 802.11 帧中，以 24 Mbps 的全速通过 5GHz/2.4GHz 电波发射出去 [v3.0 + HS]。对上层应用（如文件传输软件）来说，它根本不知道底层换了天线，它只看到传输速度飙升了近 10 倍。
**阶段 4：过河拆桥，卸载 Wi-Fi 卸套**
收尾：当 100MB 的文件全部传输完毕，L2CAP 层的发送队列清空。
回切：双方通过蓝牙控制通道发送一个 Move Channel 指令，把逻辑链路从 Wi-Fi 物理层重新切换回普通的经典蓝牙基带。
断电：为了绝对的省电，系统会立刻关闭（Power Off）Wi-Fi 硬件的射频天线，让其回到 0 功耗状态。此时，两个设备继续维持着微弱的经典蓝牙连接，等待下一次大数据任务。

>由于蓝牙 AMP 的固件要求太奇葩（必须蓝牙和 Wi-Fi 是一家芯片厂出的才能打通），手机厂商直接绕过蓝牙组织，在操作系统级别自己实现了大文件互传：
苹果生态：隔空投送 (AirDrop)。
安卓/鸿蒙生态：快传 (Quick Share)。
车联网：无线 CarPlay / Android Auto [Used for discovery and reconnection only]。

蓝牙协议栈从下到上分为三大部分：**控制器（Controller）**、**主机（Host）**和**应用层（Profiles）**。
经典的蓝牙硬件芯片（如泰凌微、Nordic、博通芯片）内部跑的是 Controller；而手机操作系统内核（如 iOS / Android / Linux BlueZ）中跑的是 Host。它们之间通过 HCI（主机控制器接口） 进行标准化通信。

### 1. Controller（控制器）= 码头上的起重机和苦力（纯硬件芯片）
它在哪里：它就是死死焊在手机主板上、或者车机电路板上的那颗蓝牙芯片物理硅片（比如北欧的 Nordic、国内的泰凌微、或者苹果御用的博通芯片）。
它干什么活：它干的是“纯体力活”和“物理学计算”。它负责控制 2.4GHz 的天线开关，把电波发射出去；或者在空气中抓取微弱的电波信号；它还要负责“跳频”（一秒钟换上百次频率，防止被地库里的 Wi-Fi 干扰）。
为什么要单独做成芯片：空气中的无线电波变幻莫测，处理速度要求在微秒（

）级别，要求极高。如果让复杂的手机操作系统（iOS/Android）亲自去数波形，手机 CPU 早就发热卡死了。所以，必须由这颗独立的硬件芯片全权包办。
### 2. Host（主机）= 港口的海关办公室（操作系统内核）
它在哪里：它不是硬件，它是纯软件代码，直接运行在手机操作系统的内核里（比如 iOS 内核、Android 的 Linux 内核）。
它干什么活：它是整个蓝牙的“智囊团”和“大管家”。它根本不关心无线电波是怎么飞的，它只关心数据的含义和安全。
我们前面刚聊到的 SSP（安全简易配对，ECDH 算密钥过程） [v2.0 / 2.1 + EDR]，就是由 Host 在手机系统里用软件算出来的。
管理蓝牙设备的各种特征值数据库（GATT）、把大文件切成小数据包（L2CAP），全都是由 Host 负责。
为什么要写在操作系统里：因为安全策略、配对弹窗、权限控制经常需要更新。把它写成软件内核代码，苹果或谷歌就能通过系统升级（比如更新到 iOS 19/20）随时修补蓝牙漏洞，而不需要你把手机拆开去换里面的蓝牙芯片。
### 3. HCI（主机控制器接口）= 港口与海关之间的标准化对讲机
它是什么：它是行业联盟（Bluetooth SIG）制定的一套全球通用的、标准化的命令语言和物理接口（比如通过串口 UART 或 USB 连线）。
为什么至关重要：因为苹果做 iOS 系统（Host），但它不自己生产所有的蓝牙芯片（Controller），它可能今年买博通的，明年买高通的。
有了标准的 HCI 接口，iOS 系统只需要通过标准的 HCI 指令（比如：发送 HCI_LE_Create_Connection 命令）喊一嗓子，底下的任何一块硬件芯片（不管是谁家生产的）就都能立刻听懂，并立刻去执行物理层面的天线对轰。

### A2DP、HFP 和 SPP
A2DP、HFP 和 SPP 是经典蓝牙（Bluetooth Classic）时代最核心的三大 应用层协议规范（Profiles）。它们运行在蓝牙协议栈的 Host（主机层） 顶端，通过标准化的数据格式，让不同的硬件（如手机和车机）能够直接听懂彼此的业务意图。
1. A2DP (Advanced Audio Distribution Profile) —— 高级音频分发规范
大白话理解：蓝牙界的“高清音乐专线”。
工作特点：专门用来传输单向、高带宽、高质量的音频流（通常是立体声）。为了保证音乐好听，它在传输前会对音频进行压缩编码（如 SBC、AAC、aptX、LDAC），发给对方后再解码播放。它只管播音乐，不管麦克风录音。
典型场景：手机向车载音响点播手机里的 QQ 音乐/网易云音乐、手机连接无线头戴式耳机听歌。
2. HFP (Hands-Free Profile) —— 免提电话规范
大白话理解：蓝牙界的“双向电话对讲专线”。
工作特点：专门用来处理双向、低带宽、低延迟的语音通话。它同时接管了音频播放（汽车喇叭）和音频输入（车顶麦克风）。为了保证通话不卡顿，它牺牲了音质（采用单声道窄带或宽带语音编码，如 CVSD、mSBC），但换取了极高的实时性，并且支持通过 AT 指令控制手机挂断、接听、查询电量。
典型场景：开车时，手机来电，你直接按下方向盘上的物理按键，通过车载麦克风和喇叭接听电话。
3. SPP (Serial Port Profile) —— 串口仿真规范
大白话理解：蓝牙界的“虚拟无限制数据线”。
工作特点：它在蓝牙底层（RFCOMM 层）上直接仿真出一条传统的 RS-232 串行物理导线。它不管你传的是音乐还是电话，它只负责把发送方的原始二进制字节（Raw Bytes），100% 原封不动地透明传输给接收方。
典型场景：
在汽车领域：早期的手机 App 连接 OBD 诊断盒子读取车辆故障码。
与 MFi 的关系：苹果为了垄断这条数据线，对标准的 SPP 进行了严格封锁。第三方设备必须内置苹果认证芯片并遵循 iAP2 协议，才能在 iOS 系统上拉起一条基于类似 SPP 机制的 外部附件会话 (EA Session)，进而传输车控数据（如宝马 G2 协议、Token 交换）。

### HCI
HCI 的全称是 Host Controller Interface（中文翻译为：主机控制器接口）。
它是蓝牙架构中最关键的一道物理和软件分界线。在硬件上，它直接决定了蓝牙芯片（Controller）如何与 CPU / 手机处理器（Host）连线通信；在软件上，它是一套标准的通信协议指令集。

## CRC4

**数据冗余校验**
1. 什么是 CRC-4？
后面的数字 4 代表校验码的长度是 4 个二进制位（4 bits）。
它意味着最终算出来的防伪标签只能在 0000 到 1111 之间。
2. 什么是多项式 10011？
在数学上它代表 `1*X4 + 0*X3 + 0*X2 + 1*X1 + 1*X0`

>CRC-4 对应的除数必然是 5 位数,因为必须把最左边的除完,要余4位的校验位置

**发送方：计算校验码并贴上标签**
假设发送方（比如你的手机蓝牙芯片）现在要发送一串 7 位数的原始数据：1101001。
步骤 1：在数据尾部补零（给标签留出空位）
因为我们要算的是 CRC-4（4位校验码），所以先在原始数据后面强行加上 4 个 0：
变成被除数：1101001 0000
步骤 2：进行“模2除法”计算余数（核心数学过程）
我们用 11010010000 除以约定好的 10011。记住，这里的减法一律换成异或运算（相同为0，不同为1）：
步骤 3：拼装发送
算出的余数是 1101，这就是我们的 CRC-4 校验码。
发送方把最后补的 4 个 0 换成 1101，组合成最终的物理数据包发射出去：
最终发送的数据：1101001 1101（前7位是数据，后4位是防伪标签）。

**接收方：如何“对答案”验证数据**
它直接用收到的整串数据，再次除以那个暗号 10011（同样使用模2除法）：
场景 A：如果余数等于 0000
结论：说明数据在空气中传输时完美无损，总指挥（LL层）可以放心收货，传给上层解密。
场景 B：如果余数不等于 0000
结论：说明路受到了地库 Wi-Fi 的电磁干扰，中间某些位变成了错的（比如 1 变成了 0）。基带层会直接把这个包丢弃，并全自动向手机申请重传（ARQ 机制），你的上层 App 甚至完全不会感知到这次丢包。

## GATT
**传统的蓝牙通信**：大部分开发者使用的是 GATT (BLE)。这就像是“发短信”或“寄挂号信”，每次只能发很小的特征值（Characteristic）数据包，效率低、限制大。

## L2CAP

L2CAP 全称是 **Logical Link Control and Adaptation Protocol**（逻辑链路控制与适配协议）,直接建立一个原生的、流式的（Streaming）数据通道。这就像是在两台设备之间拉了一根直接传输二进制数据的“水管”。

>利用`PairingOrchestrator` 负责在 BLE L2CAP 和 ExternalAccessory 两条通道之间做优先级选择、回退和切换，相当于“路由总控”。

>特点：速度比普通 BLE 快得多、延迟极低，非常适合用来传输大量数据（如固件升级 OTA、连续的传感器数据、甚至低速率音频）。

## 通道选择
通道 A (BLE L2CAP)：走低功耗蓝牙的专属高速水管。
通道 B (ExternalAccessory)：通常指 MFi（Made for iPhone）认证的通道。比如通过经典蓝牙（BR/EDR）或者 Wi-Fi/USB 连线传输。

## 两条通道的优缺点对比：
**ExternalAccessory (经典蓝牙/Wi-Fi)**：带宽极大、很稳定，但是耗电高，或者连接建立比较慢。
**BLE L2CAP**：省电、连接快，但是带宽没有前者大，且容易受到距离和无线干扰的影响。

## ExternalAccessory和Classic BT/WiFi区别

| 特性 | ExternalAccessory (EA) | 经典蓝牙 (Classic BT) | Wi-Fi |
|---|---|---|---|
| 本质是什么 | 苹果独家的 软件开发框架 | 无线 硬件通信技术 | 无线 硬件通信技术 |
| 硬件载体 | 可跑在经典蓝牙或 USB线 上 | 2.4GHz 无线频段 | 2.4G / 5G / 6GHz 无线频段 |
| 是否需要 MFi 芯片 | 是（强制需要苹果官方认证芯片） | 在 iOS 上传自定义数据必须结合 EA，故需要 | 否（走标准的 TCP/IP 网络协议，无需认证） |
| 传输速度（带宽） | 中等到高（取决于底层是蓝牙还是线） | 中等（实际约 2~3 Mbps） | 极高（可达数百 Mbps 甚至更高） |
| 功耗 | 中等到高 | 中等 | 高 |
| 典型应用场景 | CarPlay、MFi 助听器、刷卡机、车载大屏 | 蓝牙耳机音频 (A2DP)、车载免提电话 | 视频监控、网页浏览、隔空投送 (AirDrop) |

* 架构分层不同：ExternalAccessory 是应用层的软件框架；经典蓝牙与 Wi-Fi 是底层的物理通信技术。
* iOS 权限限制：在 iOS 生态中，App 无法直接通过经典蓝牙的原始 Socket 读写自定义数据，必须强制外设嵌入 MFi 芯片并通过 ExternalAccessory 框架“套壳”才能通信。
* Wi-Fi 免认证：通过 Wi-Fi 进行数据传输时，由于走的是通用的 TCP/IP 标准网络协议栈，苹果允许直接使用原生的网络套接字（如 Network 框架、WebSocket）进行通信，无法也不需要通过 ExternalAccessory 框架来管理。



## GATT
Generic Attribute Profile（中文通常翻译为通用属性配置文件

1. **GATT 的核心工作模式（C/S 架构）**
GATT 规定了通信双方的两个角色 [1]：
GATT Server（服务端）：保存数据的设备（通常是外设/硬件，比如心率手环、温度计）。它里面存有传感器数据。
GATT Client（客户端）：向服务端发起读写请求的设备（通常是手机/App）。它去读取手环里的数据，或者向手环发送控制指令。
2. **GATT 的数据组织结构（树状层级）**
GATT 内部的数据不是乱成一团的，而是像电脑里的文件夹和文件一样，分为三个层级：
Profile（配置文件）：最高层。它并不是一个真实存在的数据块，而是一个概念集合。比如“心率配置文件”规定了一个心率手环应该包含哪些服务。
Service（服务）：相当于“文件夹”。一个设备可以有多个服务。例如：
设备信息服务（包含制造厂商、电量等文件夹）
心率服务（包含心率数据文件夹）
Characteristic（特征值）：相当于具体的“文件”，是真正存放数据的地方 [1]。比如心率服务文件夹下，有一个名叫“当前心率”的文件，里面存着 80 这个数字。
Descriptor（描述符）：相当于文件的“属性/备注”（附属于特征值）。比如标明这个心率数据的单位是 bpm，或者用来开启/关闭通知。
3. **GATT 的四种核心数据交互方式**
手机 App（Client）和蓝牙硬件（Server）通过 GATT 只能做这四件事：
Read（读）：手机主动去读取硬件某个特征值里的数据。
Write（写）：手机主动向硬件某个特征值写入数据（比如发送开灯指令）。
Notify（通知）：硬件数据变动时，自动发送给手机，手机不需要回复确认（速度快）。
Indicate（指示）：硬件数据变动时，自动发送给手机，但手机必须回复一个收到确认（更安全，但慢）。

>如果把 L2CAP 比作两台设备之间连了一根无标签的、纯流质的“水管”（你得自己定义流过去的是啥），那么 GATT 就是一个排满抽屉的储物柜。每个抽屉都有唯一的编号（UUID），手机过去翻特定抽屉的资料（Read）或者往抽屉里塞纸条（Write）。

## Questions

1. 为什么小的指令不直接使用GATT

**在 iOS/Android 的系统底层，GATT 只有一个主队列。**
如果你的 App 或系统其他组件正在通过 GATT 读取一个比较大的数据（比如同步时间、读取大日志），或者正在进行连续的 Write 操作。
此时你突然想发一个“几个字节的控制指令”，这个指令就必须在 GATT 队列里排队，导致高延迟。
L2CAP 是一个独立的、专属的管道（基于 PSM 端口）。它不和 GATT 挤同一条非确定性的单车道，能保证指令的绝对实时性。

**统一的数据封包流（Stream）与状态机**
GATT 是“块”结构：它是基于属性（Attribute）的，每次读写都需要处理 ATT MTU 限制、特征值通知（Notify/Indicate）的应答机制。
L2CAP 是“流”结构（Stream）：它和 ExternalAccessory（EA 框架）的本质非常像，两者在代码层面都可以被抽象为标准的 NSInputStream / NSOutputStream（输入输出流）。
架构设计的优雅性：对于指挥官 PairingOrchestrator 来说，如果底层都是“流”，它就可以用同一套数据封包/解包逻辑（如统一的 Protobuf 序列化）来同时对接 L2CAP 和 EA。如果强行加入 GATT，就必须为 GATT 单独写一套基于特征值的碎片化拼包逻辑，增加了架构的复杂度。

**MTU（最大传输单元）的动态优势**
GATT 的 MTU 限制较多：虽然现在 BLE 5.x 支持大 MTU，但在 iOS 上，GATT MTU 往往被系统限制在 185 字节左右。一旦控制指令偶尔变大（比如包含了一串复杂的加密认证 Token），GATT 就必须在应用层做拆包。
L2CAP 的 MTU 极大：L2CAP CoC（面向连接的通道）允许的 MTU 通常可以达到 64KB。即使是发小指令，它也具备极高的扩展性，完全不需要担心数据偶尔超限的问题。

**连接与握手开销**
在物联网或车载场景中，设备连接成功后，通常需要立即进行安全握手或认证（Pairing/Auth）。
如果走 GATT，需要先经历：发现服务（Discover Services） -> 发现特征值（Discover Characteristics） -> 开启通知（Set Notify），这一套系统轮询下来可能需要几百毫秒甚至上秒。
如果走 L2CAP，一旦 BLE 连接建立，直接通过固定的 PSM（Protocol/Service Multiplexer）端口建立连接，跳过了繁琐的 GATT 服务发现流程，直接就能互发数据。
