Digital Key Applet 存储区就在安全芯片（SE）内部，属于芯片安全区最核心、最机密的存储区域。

它不是手机普通闪存（NAND Flash）中的一个文件夹，也不是手机内存（RAM）中的一段空间，而是完全隔离在安全硬件内部的受保护非易失性存储（Secure Non-Volatile Memory，如 EEPROM 或 Flash）中。

## Part A：Digital Key Applet 存储区与核心私钥

### 一、它在硬件中的具体位置

手机或智能手表中有一颗独立的、通过了高级别安全认证（如 CC EAL6+ 金融级认证）的芯片，叫做 eSE（Embedded Secure Element，嵌入式安全元件）。
在 eSE 芯片内部，存储空间被严格划分为两部分：

- COS（Card Operating System）系统区：存放安全芯片的底层操作系统代码。
- 应用与数据区（Applet & Data Area）：这就是 Digital Key Applet 存储区。它与手机银行卡（Apple Pay/华为支付）、交通卡（公交卡 Applet）并列存在。

```text
+-------------------------------------------------------------+

|                     智能手机 (Smart Phone)                   |
|  +-------------------------------------------------------+  |
|  |             主操作系统 (Android / iOS)                |  |
|  +-------------------------------------------------------+  |
|                             │ (隔离总线)                     |
|                             ▼                               |
|  +-------------------------------------------------------+  |
|  |             安全芯片 (Secure Element, eSE)            |  |
|  |  +-------------------------------------------------+  |  |
|  |  |          芯片操作系统 (COS - Card OS)            |  |  |
|  |  +-------------------------------------------------+  |  |
|  |  +-------------------------------------------------+  |  |
|  |  |         [核心安全区] 应用与数据存储区            |  |  |
|  |  |  +-------------------------------------------+  |  |  |
|  |  |  |  银行卡 Applet (如 Apple Pay)             |  |  |  |
|  |  |  +-------------------------------------------+  |  |  |
|  |  |  |  公交卡 Applet                           |  |  |  |
|  |  |  +-------------------------------------------+  |  |  |
|  |  |  |  ★ Digital Key Applet (CCC车钥匙存储区)  |  |  |  |
|  |  |  |  ├─ 运行代码 (Applet Code)                |  |  |  |
|  |  |  |  ├─ 核心私钥 (Device Private Key)        |  |  |  |
|  |  |  |  └─ 凭证数据 (VIN / ACL / 签名证书)        |  |  |  |
|  |  |  +-------------------------------------------+  |  |  |
|  |  +-------------------------------------------------+  |  |
|  +-------------------------------------------------------+  |
+-------------------------------------------------------------+
```

### 二、该存储区域的三大“硬核”安全特性

由于位于安全芯片的安全区内，该存储区具备以下普通手机存储无法比拟的特性：

1. 物理隔离与访问受限（Hardware Isolation）
   手机的主操作系统（Android 或 iOS）没有任何权限直接读取或修改这个存储区的内容。
   只有当手机调用特定的安全指令，且通过了 SE 内部的权限校验时，SE 底层的 COS 才会允许 Digital Key Applet 运行并读取其内部的 VIN 码或 ACL。

2. 防物理克隆与侧信道攻击（Anti-Tampering）
   如果黑客试图通过物理手段（如激光切片、微探针探查芯片内部电路）去读取存储区中的私钥，SE 芯片的硬件防御机制（如光传感器、电压侦测、网格保护层）会瞬间触发，直接物理自毁（擦除所有密钥数据）。

3. 运行时内存隔离（Firewall 机制）
   基于 GlobalPlatform（GP）规范，SE 内部建有硬件防火墙。
   即使手机里的银行卡 Applet 或公交卡 Applet 遭遇了极罕见的漏洞攻击，它们在 SE 内部也绝对无法越权跨过防火墙去窥探或修改隔壁 Digital Key Applet 存储区里的车钥匙数据。

### 三、补充：它与 Android 的 TEE（可信执行环境）有什么区别？

在 Android 手机开发中，经常会听到 TEE（Trusted Execution Environment） 和 SE（Secure Element）。它们都是安全区，但车钥匙为什么要存在 SE 里？

- TEE（如 Keystore / StrongBox）：主 CPU 划分出的可信执行区，和主系统共享部分硬件资源，抗物理攻击能力相对较弱，常用于生物识别和软件密钥保护。
- SE（安全芯片）：独立硬件安全域。CCC 标准要求数字钥匙 Applet 与核心私钥存放在 SE 内，以获得更高等级的抗克隆与抗物理提取能力。

### 四、Digital Key Applet 内部的核心私钥

在 CCC（车联网联盟）标准架构中，存储在 Digital Key Applet 内部的核心私钥（Device Private Key，设备私钥），是整个车钥匙安全生态的“定海神针”。
它是数字钥匙小应用（Applet）在安全芯片（SE）内部本地原生生成的一段非对称加密算法私钥（通常基于高强度的椭圆曲线密码学，如 ECC NIST P-256 / secp256r1 或 Brainpool P256r1 曲线）[1]。

#### 1) 它是怎么来的？（诞生过程）

这个私钥绝不是车企云端生成的，也绝不是手机厂商云端生成的，更不是在生产手机时写死的。
它的诞生过程遵循极其严格的安全规范：

- 指令触发：在用户手动确认添加车钥匙的流程中，手机厂商云端（Device OEM Server）会向手机 SE 芯片内的 Digital Key Applet 发送一条“生成密钥对”的指令。
- 硬件闭门生成：SE 芯片内部的真随机数发生器（TRNG）和加密硬件加速器在芯片内部瞬间计算，直接生成一对非对称密钥对（KeyPair）：
  - 设备公钥（Device Public Key）：被导出来，传回给车企云端，用来制作凭证和证书。
  - 设备私钥（Device Private Key）：被永久锁死在 SE 芯片的这一块 Applet 存储区内，永不离芯。

#### 2) 它在日常解锁中扮演什么角色？（它的工作原理）

黑客可以通过无线电去监听手机和汽车之间的蓝牙或 NFC 通信。如果手机只是简单地把 VIN 码或者证书发送给汽车，黑客录制下这段无线电信号，稍后就能“重放（Replay Attack）”来偷车。
为了防止这种情况，核心私钥在解锁时会执行数字签名（Challenge-Response，质询-应答机制）：

```text
 [汽车端 (Vehicle)]                                    [手机安全芯片 Applet (SE)]
        │                                                       │
        │ 1. 发起质询：发出一串单次有效的随机数 (Nonce)         │
        ├──────────────────────────────────────────────────────►│
        │                                                       │ 2. 闭门计算：
        │                                                       │    Applet 调用内部的
        │                                                       │    【核心私钥 (Private Key)】
        │                                                       │    对这个随机数进行数字签名
        │                                                       │
        │ 3. 应答：返回签名结果 (Signature)                    │
        ◄───────────────────────────────────────────────────────┤
        │                                                       │
        │ 4. 车端校验：
        │    车端使用事先存储的【设备公钥】验证签名。
        │    如果对得上，证明手机确实拥有合法的私钥。
        ▼
 (车门解开！)
```

## Part B：CCC 标准架构与通信机制

### 五、什么是 CCC (Car Connectivity Consortium) 标准架构

CCC（Car Connectivity Consortium，车联网联盟）标准架构是全球汽车与消费电子行业共同制定的一套跨平台、高安全性的数字钥匙（Digital Key）标准化生态系统。它旨在打破手机品牌与汽车品牌之间的壁垒，确保车主仅通过智能手机或可穿戴设备，就能实现无缝且安全的无钥匙进入、解锁、启动及钥匙分享功能。

CCC 标准架构并非单一的技术，而是一个由四大核心实体、三大无线通信层、以及高强度安全芯片（SE）与公钥基础设施（PKI）共同构成的端到端信任链条。

#### 1) 架构的四大核心实体

CCC 架构通过标准化接口，将车辆、手机及其各自的云端服务器连接在一起，形成一个闭环：

```text
+-------------------------+                   +-------------------------+

|   Vehicle OEM Server    | <== 标准化接口 => |   Device OEM Server     |
|      (汽车厂商云端)      |                   |     (手机厂商云端)      |
+-------------------------+                   +-------------------------+
             ^                                             ^

             | 专有/标准化接口                             | 专有/标准化接口
             v                                             v
+-------------------------+                   +-------------------------+

|        Vehicle          | <== 三层无线通信 => |      Mobile Device      |
|         (车辆)          |   (UWB / BLE / NFC) |   (手机 / 安全芯片 SE)  |
+-------------------------+                   +-------------------------+
```

移动设备 (Mobile Device)：车主使用的智能手机或手表。其内部核心是安全元件（Secure Element, SE），用于安全地创建、存储和处理数字钥匙相关凭证，与操作系统隔离，防止软件和硬件攻击。

车辆 (Vehicle)：集成了 NFC 读卡器、蓝牙（BLE）和超宽带（UWB）天线的汽车。车辆内部存储有对应信任链的公钥。

车辆厂商服务器 (Vehicle OEM Server)：管理车辆生命周期、车主配对状态及高级钥匙分发策略的云端。

设备厂商服务器 (Device OEM Server)：如 Apple、Samsung 等手机厂商的云端，负责与手机 SE 安全芯片对接，协助将数字钥匙安全地下发到手机中。

#### 2) 三层无线通信技术架构

在最新的 CCC Release 3 / 4 规范中，车辆与手机之间的近场交互采用了由远及近的“三层防御与感知”架构：

BLE（低功耗蓝牙）—— 远程唤醒与身份验证：当手机靠近车辆（约数十米）时，BLE 首先建立连接，完成设备身份的初步认证并唤醒车辆系统。

UWB（超宽带）—— 精准定位与迎宾（无感进入）：当距离进一步缩短，UWB 利用其厘米级的定位精度进行安全测距。它能精准判断车主是在车外还是车内，并在防范“中继攻击（Relay Attack）”的同时，实现自动解锁、亮起迎宾灯等无感体验。

NFC（近场通信）—— 触碰解锁与低电量备用：用于刷车把手解锁或放置在车内无线充电板上启动车辆。其最大优势是支持低电量模式，即使手机因没电关机，依然可以通过 NFC 芯片的残存电量刷卡开门和发动汽车。

#### 3) 安全与信任机制（PKI 架构）

CCC 架构之所以能达到金融级的安全，依赖于其底层的公钥基础设施（PKI）和非对称加密算法：

端到端信任链：数字钥匙在手机的 SE 芯片中生成，其私钥绝不离设备。车辆仅存储相应的公钥。

相互签名认证：每次解锁时，车辆和手机通过非对称密码技术进行双向签名校验，只有当签名计算完全吻合，车辆才会执行解闭锁或启动发动机指令。

安全合规：CCC 数字钥匙的 Applet（小应用）通过了高等级的通用准则（Common Criteria, CC）安全评估，确保能有效抵御存储入侵和硬件克隆。

#### 4) 核心业务场景流程

车主配对（Owner Pairing）：车主通过交付中心拿到的机密凭证，在车内将自己的手机与汽车绑定。手机云端与车企云端握手，将数字钥匙安全地写入手机 SE 中。

钥匙分享（Key Sharing）：车主无需物理交付，可以直接通过 IM 软件（如微信、iMessage）向好友发送一个加密分享链接。在云端校验证书链后，“朋友设备”也能安全地获得一个受限的数字钥匙（可自定义使用时限或功能权限）。

#### 5) 总结

CCC 标准架构通过“云端（厂商云对云）+ 管道（BLE/UWB/NFC）+ 终端（手机 SE 与汽车硬件）”的联合标准化，建立了一套全球通用的数字钥匙技术规范。它成功地让手机彻底取代传统物理钥匙，同时保障了极高的安全性和一致的用户体验。

## Part C：常见问题（Q&A）

### Q1：上图架构中，Device OEM Server 怎么和手机/安全芯片 SE 关联？

在 CCC 标准架构中，Device OEM Server（设备厂商云端，如苹果/三星/华为云） 与手机终端的 安全芯片 SE（Secure Element） 之间，是通过一条专有的、端到端的安全硬核通道进行关联的。

在技术实现上，这条通道遵循 GlobalPlatform（GP）标准规范，其核心关联和通信机制可以拆解为以下四个层面：

#### 1) 核心关联纽带：SPMS / TSM 架构

设备厂商服务器内部集成了 SPMS（Secure Provisioning Management System，安全配置管理系统） 或传统的 TSM（Trusted Service Manager，可信服务管理器） 功能。

- 唯一性绑定：手机在出厂时，其内置的 eSE（嵌入式安全芯片）就已经烧录了由设备厂商生成的、全球唯一的芯片根证书和密钥对（如公钥/私钥）。
- 云端建档：Device OEM Server 的数据库中记录了该设备 SE 的唯一样本标识（如 00A1... 序列号）。当手机激活并绑定账号后，云端服务器便在逻辑上将“云端账号-手机系统-硬件 SE”三者强关联在一起。

#### 2) 数据的安全传输通路：端到端加密

当汽车厂商云端（Vehicle OEM Server）通过标准接口，把生成的数字钥匙凭证（Credential）同步给 Device OEM Server 后，两者的通信路线如下：

```text
[Device OEM Server]
        │
        │ (1) 通过 TLS 建立互联网安全连接
        ▼
[Mobile Phone OS / Wallet App] (手机主系统 / 钱包应用)
        │
        │ (2) 使用专有 APDU 指令透传
        ▼
[Secure Element (SE)] ──► [Digital Key Applet] (数字钥匙小应用)
```

云端至手机（网络层）：Device OEM Server 与手机的操作系统（如 iOS / Android）建立高强度的 TLS 加密网络连接，将数据下发给系统自带的“钱包（Wallet）”或核心系统服务。

手机内部至 SE（硬件层）：主系统（AP，应用处理器）通过手机内部的总线接口（如 SPI / I2C），使用标准的 APDU（应用协议数据单元）指令，将云端数据包原封不动地“透传（Pass-through）”给 SE 芯片。

注意：手机主系统（Android/iOS）在整个过程中只是一个“搬运工”，它无法解密和窥探这个数据包的内容，从而防止了手机因越狱、root 或中恶意软件而导致鑰匙泄露。

#### 3) 钥匙在 SE 内部的落地：SCP 加密安全通道

当数据包到达 SE 内部后，SE 会使用 SCP（Secure Channel Protocol，安全通道协议，通常为 SCP03/SCP11 规范） 与 Device OEM Server 进行最终的硬件级身份核验。

- 解密与验签：数据包使用只有 Device OEM Server 和该 SE 芯片知道的专有对称密钥（Session Key）进行了加密。SE 内部的硬件加密引擎会对数据进行解密，并验证 Device OEM Server 的数字签名。
- 写入 Applet：确认数据来源合法且未被篡改后，SE 会将数字钥匙写入专用的 Digital Key Applet（数字钥匙小应用） 存储区中。至此，数字钥匙正式在手机硬件底层“安家”。

#### 4) 总结：为什么要通过 Device OEM 绕这一圈？

之所以不让汽车厂商服务器（Vehicle OEM Server）直接控制手机 SE，是因为 SE 的最高控制权（主控密钥）掌握在手机硬件厂商（Device OEM）手里。
通过这种架构，汽车厂商只需要和手机厂商的云端对接（标准接口），手机厂商利用自己对自家硬件（SE）的绝对控制权，安全、合规地把钥匙送入硬件底层。这在保证金融级安全性的同时，极大地降低了车企与各类手机进行硬件适配的复杂度。

### Q2：(1) 通过 TLS 建立互联网安全连接这一步怎么实现？需要我自己开发的 App 做什么？

在 CCC 标准架构中，“Device OEM Server（手机厂商云） 与 手机系统/钱包 App 之间通过 TLS 建立互联网安全连接” 这一步，100% 是由手机操作系统（如 iOS / Android）和手机厂商的底层系统服务（系统级原生钱包）自动处理的，完全不需要、也不允许你开发的第三方 App 去实现或建立。
第三方 App 的核心职责只是作为一个引发者（Trigger）和中间状态展示者。以下是其底层的实现逻辑以及你需要做的具体开发工作：

#### 一、这一步在系统底层是如何实现的？

当你开发的汽车品牌 App 发起数字钥匙请求时，真正的底层通信是由系统级原生服务接管的：

- 系统级接管：你的 App 调用手机系统的数字钥匙 SDK（例如 iOS 的 PassKit 框架，或 Android 的 Google Wallet API / 华为 Identity Kit）。
- 建立安全连接：操作系统（而非你的 App）在后台自动与对应的手机厂商服务器（如苹果的 Apple Pay 服务器，华为、小米的钱包服务器）建立专有的、双向认证的 TLS 1.3 / 密评安全通道。
- 身份凭证校验：手机厂商服务器通过该 TLS 通道下发配置指令，验证当前手机的设备证书（Device Certificate），确保这台手机是合法的、非仿冒的设备。

#### 二、你自己开发的 App 需要做什么？

你开发的 App 在这个阶段扮演的是“发起人”和“数据传递桥梁”的角色。你需要完成以下四步核心工作：

```text
[你开发的汽车 App] ──(1. 请求钥匙数据)──► [汽车厂商云端 (Vehicle OEM Server)]
       │                                                   │
       │                                        (2. 云对云下发标准凭证)
       ▼                                                   ▼
[调用手机系统 SDK]                                [设备厂商云端 (Device OEM Server)]
       │                                                   │
       │ (3. 系统接管: 建立 TLS)                            │
       └─────────────────────────► [手机系统钱包] ◄─────────┘
```

1. 向汽车厂商云端请求钥匙数据包（Opaque Token / Payload）
   你的 App 不能直接生成钥匙。当用户点击“激活车钥匙”时，你的 App 必须通过 API 向你们自己的汽车厂商服务器（Vehicle OEM Server）发起请求，申请一个加密的、未初始化的数字钥匙数据包（通常称为凭证令牌 Token 或 Payload）。

2. 调用手机操作系统提供的数字钥匙原生 SDK
   获取到汽车云端的数据包后，你的 App 需要调用手机厂商官方的 SDK 接口。

iOS (Swift / Objective-C)：使用 PassKit 框架，构建一个 PKAddPassButton，并准备好 PKSecureElementPass 的数据，调用系统的添加卡片界面。

Android (Java / Kotlin)：调用 Google Wallet SDK 中的 savePasses 方法，或国内手机厂商（如华为、小米）提供的专有车钥匙服务 SDK。

3. 将控制权完全移交给手机系统
   一旦你调用了系统 SDK，你的 App 必须交出控制权。此时，手机屏幕上会弹出一个由系统控制的、无法被篡改的“钱包添加车钥匙”原生界面（例如苹果的经典添加卡片弹窗）。在这个界面弹窗期间，所有的 TLS 通信、安全加密、SE 芯片写入，都由系统在后台独立完成，你的 App 无法干预，也无法读取任何传输中的数据。

4. 监听回调并刷新 App 界面状态
   系统钱包处理完 TLS 连接并成功将钥匙写入 SE 芯片后，系统 SDK 会给你的 App 返回一个结果回调（Callback）：

如果成功（Success）：你的 App 需要更新 UI（例如从“未激活”变为“已激活”），允许用户在 App 内看到钥匙状态，并引导用户如何使用（如：将手机靠近车把手刷卡）。

如果失败（Failure）：你的 App 需要捕获错误码（如网络超时、SE 空间不足、用户取消等），并向用户展示友好的错误提示。

#### 三、核心开发伪代码示例

以 iOS (Swift) 调用系统原生钱包添加车钥匙为例，你的 App 内部逻辑通常非常精简，如下所示：

```swift
// 1. 假设这是你从汽车厂商服务器拿到的加密凭证数据
let cardPayloadFromServer = "xxxx_from_vehicle_oem_server_xxxx"

// 2. 初始化系统级车钥匙配置对象
guard let passRequest = PKAddSecureElementPassConfiguration(
    issuerIdentifier: "your.car.oem.id",
    credentialIdentifier: cardPayloadFromServer
) else { return }

// 3. 实例化系统添加界面控制器（此时系统接管，开始进行底层的 TLS 通信与 SE 写入）
guard let addPassNavController = PKAddSecureElementPassViewController(configuration: passRequest, delegate: self) else { return }

// 4. 弹出系统界面
self.present(addPassNavController, animated: true, completion: nil)

// 5. 编写代理回调方法，监听系统处理结果
func addSecureElementPassViewController(_ controller: PKAddSecureElementPassViewController,
                                        didFinishWith passes: [PKSecureElementPass]?,
                                        error: Error?) {
    controller.dismiss(animated: true) {
        if let error = error {
            // 处理失败：可能 TLS 连接断开、或者 SE 写入失败
            print("添加失败: \(error.localizedDescription)")
        } else {
            // 处理成功：钥匙已在 SE 安全芯片落地，通知汽车云端并刷新 App UI
            print("车钥匙成功写入 SE 安全芯片！")
        }
    }
}
```

### Q3：写入 Applet 数据是啥，是一个证书么？

在 CCC 标准架构中，写入 Digital Key Applet（数字钥匙小应用）的数据不仅仅是一个证书（Certificate），而是一个包含了“独立运行程序”与“多种密钥、证书凭证”的完整“安全数据包裹” [1, 2]。
因为安全芯片 SE（Secure Element）其实是一个微型的计算机硬件（有它自己的 CPU 和安全操作系统），所以写入的过程可以理解为“先安装一个小程序，再往里面存入账号密码和通关护照”。
这个被写入的数据包裹，在技术上主要包含以下三个核心部分：

#### 一、写入数据的三大核心组成

1. 小程序代码（Digital Key Applet Code）
   它是什么：这是一段符合 Java Card 规范的编译好的字节码（Bytecode） [1, 2]。
   它的作用：它规定了车钥匙在硬件底层该如何运行。比如，当汽车通过 NFC 或蓝牙发来一段指令时，这个小程序会控制 SE 芯片去执行对应的非对称加密算法，计算并返回结果。

2. 设备专属密钥对（Device Key Pair）
   它是什么：这是在安全芯片（SE）内部原生生成或安全注入的一对非对称加密密钥（包含一个私钥 Private Key 和一个公钥 Public Key）。
   它的作用：私钥绝对不会离开 SE 芯片，任何人（包括手机厂商和你的 App）都读不出来 [1, 2]。每次你靠近汽车解锁时，Applet 小程序就会调用这个私钥去对汽车发来的随机数进行数字签名。

3. 证书链与凭证（Certificate Chain & Credentials）—— 你提到的证书就在这里
   这才是你提到的“证书”部分，它用于向汽车证明这把钥匙是合法合规的：

Device OEM 签名证书：证明这个 SE 芯片是正品（比如证明这是苹果或华为官方生产的合格安全芯片）。

车企签名证书（Vehicle OEM Digital Signature）：证明这把钥匙已经被车企云端认可，并且已经和你的某台具体车架号（VIN）绑定 [1]。

访问控制列表（ACL）：里面记录了这把钥匙的权限信息。例如：是车主钥匙还是分享出去的临时钥匙？是否允许启动发动机？它的有效期到什么时候？

#### 二、为什么不只写入一个证书？

如果只写入一个证书，黑客只需要通过软件手段把这个证书文件“复制、拷贝”到另一台手机上，就能偷走汽车。
而 CCC 标准写入 Applet 数据，是因为证书必须配合“出不来、防克隆”的硬件私钥以及安全小程序的加解密计算逻辑（Applet Code），才能确保这把数字钥匙在物理层面上是天下无双、无法被复制的 [2]。

### Q4：VIN 号和 ACL 在车企签名证书上么？

在 CCC（车联网联盟）标准架构中，VIN 码（车架号）和 ACL（访问控制列表）并不直接包含在车企的“签名证书”里，而是存储在由车企签名保护的“凭证结构体（Credential Structure / Payload）”中。
在密码学和 CCC 规范中，证书（Certificate）和凭证/属性数据（Credential/Attributes）是相互独立但紧密结合的两个概念。

#### 一、数据是如何组织的？（证书 vs 凭证）

如果把车钥匙比作一张“出入证”，它们的层级结构是这样的：

1. 车辆凭证数据包裹（Credential Info）—— 包含 VIN 和 ACL
   这是车企云端（Vehicle OEM Server）生成的核心数据包，它是一个结构化的数据块（通常遵循 ASN.1 编码规范），里面明确记录了：

车辆身份：VIN 码（车架号）或车辆的唯一标识符（Vehicle Identifier）。

权限规则：ACL（访问控制列表），例如：允许启动发动机、允许开启后备箱、钥匙有效期（Time Windows）、是否允许分享等。

手机公钥：该手机 SE 芯片为这把钥匙生成的专属公钥（Device Public Key）。

2. 车企签名（Vehicle OEM Signature）—— 盖章保护
   车企云端使用自己的私钥（Vehicle OEM Private Key），对上述包含 VIN 和 ACL 的“凭证数据包裹”进行数字签名。
   这个签名就像一个防伪火漆印。任何人都无法修改里面的 VIN 码或放宽 ACL 权限，一旦修改，签名立即失效。

3. 车企签名证书（Vehicle OEM Certificate）—— 用于验证签名
   车企的签名证书是一个标准的 X.509 证书。它里面不包含特定某辆车的 VIN 码，而是包含：

车企的身份信息（如：MyApp, BYD, Tesla 等）。

车企的公钥（Vehicle OEM Public Key）。

#### 二、为什么不直接写在车企证书里？

在数字证书体系中，证书通常是长期有效且大范围通用的。
如果把每辆车的 VIN 码和经常变动的 ACL（比如租车给别人，ACL 有效期只有 2 天）写在车企的根证书或二级证书里，会导致车企需要为每一把钥匙、每一次权限修改都去发行一张全新的 CA 证书。
这会导致证书链变得无比臃肿，严重消耗手机 SE 芯片和车端安全芯片（Secure Element）本就极度珍贵的存储空间和计算算力。
因此，CCC 采用的做法是：证书保持通用，数据（VIN + ACL）保持独立，通过“签名”将证书与数据强行绑定。

#### 三、车端解锁时的真实校验流程

当手机靠近汽车时，车端的小程序会按照以下逻辑去读出并校验这些数据：
读证书：车端先读取手机发过来的 车企签名证书，用车辆出厂时内置的根证书去验证它，确认“这确实是我们公司官方发出的证书”。
验签名：车端利用证书里的车企公钥，去解密和校验手机发来的凭证数据包裹的签名。如果签名通过，说明里面的内容 100% 真实，未被篡改。
对 VIN 码：车端检查凭证里写着的 VIN 码 是否与当前这辆车自己的 VIN 码完全一致。如果对不上，说明这把钥匙是隔壁邻居车的，拒绝解锁。
查 ACL 权限：车端比对当前的系统时间。如果当前时间在 ACL 规定的有效期内，并且车主正在尝试的操作（如开启后备箱）在 ACL 允许的列表里，车辆才会最终执行动作。

#### 总结

VIN 码和 ACL：在凭证内容（Credential / Payload）中。
车企签名证书：只负责提供验证公钥。
它们共同被安全写入到手机的 Digital Key Applet 存储区中。
