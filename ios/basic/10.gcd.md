# GCD 线程调度原理与使用场景

---

## 一、GCD 调度核心原理

### 1.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         应用层                                   │
│   dispatch_async / dispatch_sync / dispatch_after / ...        │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                      队列层 (Queue Layer)                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │主队列(Main)│ │串行队列  │ │并发队列  │ │全局队列  │          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                    调度层 (Scheduling Layer)                     │
│  - 任务排队 (FIFO/优先级)                                        │
│  - 线程池管理                                                    │
│  - QoS 映射                                                      │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                     内核层 (Kernel Layer)                        │
│  - pthread 线程池                                                │
│  - mach semaphore 唤醒机制                                       │
│  - kqueue 事件驱动                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心调度机制

| 机制 | 说明 |
|------|------|
| **生产者-消费者模型** | `dispatch_async` 是生产者，工作线程是消费者 |
| **信号量同步** | 使用 mach semaphore 实现线程等待/唤醒 |
| **工作窃取** | 空闲线程可从其他队列"偷"任务 |
| **QoS 优先级** | 高优先级任务抢占资源 |

```c
// 核心调度伪代码
void dispatch_async(queue, block) {
    enqueue(block, queue);           // 1. 任务入队
    
    if (needs_thread(queue)) {        // 2. 判断是否需要线程
        thread = create_or_wakeup_thread();
    }
    
    signal_thread(thread);            // 3. 唤醒线程执行
}
```

## 二、GCD 的使用场景详解

### 2.1 主线程场景

#### 场景1：UI 更新（必须）

```swift
// ✅ 正确：UI 更新必须回到主线程
DispatchQueue.main.async {
    self.label.text = "更新完成"
    self.activityIndicator.stopAnimating()
}

// ❌ 错误：子线程更新 UI 会崩溃
DispatchQueue.global().async {
    self.label.text = "这行会崩溃"
}
```

#### 场景2：延迟执行

```swift
// 延迟 2 秒在主线程执行
DispatchQueue.main.asyncAfter(deadline: .now() + 2) {
    self.showToast("网络超时")
}

// 使用场景：启动页停留、提示消失、防抖
```

#### 场景3：保证线程安全的数据访问

```swift
class UserManager {
    private var user: User?
    private let queue = DispatchQueue.main
    
    func updateUser(_ user: User) {
        // 确保在主线程访问，避免多线程问题
        queue.async {
            self.user = user
            self.notifyObservers()
        }
    }
}
```

### 2.2 全局队列场景

#### 场景1：网络请求与数据处理

```swift
class NetworkService {
    func fetchUser(completion: @escaping (User?) -> Void) {
        // 使用 .default 优先级（全局队列默认）
        DispatchQueue.global().async {
            // 1. 网络请求
            let data = self.performNetworkRequest()
            
            // 2. JSON 解析
            let user = self.parseJSON(data)
            
            // 3. 回到主线程更新 UI
            DispatchQueue.main.async {
                completion(user)
            }
        }
    }
}
```

#### 场景2：按优先级区分任务

```swift
// 高优先级：用户正在等待的结果
DispatchQueue.global(qos: .userInteractive).async {
    // 处理用户拖拽、动画相关计算
}

// 较高优先级：用户发起的操作
DispatchQueue.global(qos: .userInitiated).async {
    // 打开文档、加载用户请求的数据
}

// 默认优先级：普通任务
DispatchQueue.global(qos: .default).async {
    // 普通业务逻辑
}

// 低优先级：可延迟的任务
DispatchQueue.global(qos: .utility).async {
    // 下载图片、数据同步
}

// 后台优先级：用户不感知的任务
DispatchQueue.global(qos: .background).async {
    // 预加载、缓存清理、日志上报
}
```

#### 场景3：并发执行多个独立任务

```swift
// 并发下载多张图片
let group = DispatchGroup()

for url in imageURLs {
    DispatchQueue.global().async(group: group) {
        let image = downloadImage(from: url)
        cacheImage(image, for: url)
    }
}

group.notify(queue: .main) {
    print("所有图片下载完成")
    self.collectionView.reloadData()
}
```

### 2.3 自定义串行队列场景

#### 场景1：保护共享资源（替代锁）

```swift
class BankAccount {
    private var balance: Double = 0
    private let serialQueue = DispatchQueue(label: "bank.serial")
    
    func deposit(_ amount: Double) {
        serialQueue.async {
            self.balance += amount
        }
    }
    
    func withdraw(_ amount: Double) -> Bool {
        var success = false
        serialQueue.sync {
            if self.balance >= amount {
                self.balance -= amount
                success = true
            }
        }
        return success
    }
}
```

#### 场景2：确保任务顺序执行

```swift
class AnimationQueue {
    private let serialQueue = DispatchQueue(label: "animation.serial")
    
    func animate(in sequence: [(UIView, () -> Void)]) {
        var idx = 0
        func next() {
            guard idx < sequence.count else { return }
            let (view, animation) = sequence[idx]
            idx += 1
            
            // 动画完成后执行下一个
            UIView.animate(withDuration: 0.3, animations: animation) { _ in
                self.serialQueue.async {
                    next()
                }
            }
        }
        
        serialQueue.async {
            next()
        }
    }
}
```

#### 场景3：解决读写锁问题

```swift
class ThreadSafeDictionary<K: Hashable, V> {
    private var dict: [K: V] = [:]
    private let concurrentQueue = DispatchQueue(label: "dictionary.concurrent", attributes: .concurrent)
    
    // 读：可以并发
    func value(for key: K) -> V? {
        return concurrentQueue.sync {
            return dict[key]
        }
    }
    
    // 写：需要 barrier 确保互斥
    func setValue(_ value: V, for key: K) {
        concurrentQueue.async(flags: .barrier) {
            self.dict[key] = value
        }
    }
}
```

### 2.4 自定义并发队列场景

#### 场景1：限制并发数量的任务处理

```swift
// 限制最多同时处理 3 个任务
let concurrentQueue = DispatchQueue(label: "download.concurrent", 
                                     attributes: .concurrent)

let semaphore = DispatchSemaphore(value: 3)

for url in downloadURLs {
    concurrentQueue.async {
        semaphore.wait()
        downloadFile(from: url)
        semaphore.signal()
    }
}
```

#### 场景2：批量数据处理

```swift
// 并发处理数组元素
let numbers = Array(1...10000)
let resultQueue = DispatchQueue(label: "result.concurrent", attributes: .concurrent)
var results: [Int] = []
let lock = NSLock()

DispatchQueue.concurrentPerform(iterations: numbers.count) { index in
    let result = heavyComputation(numbers[index])
    
    resultQueue.async(flags: .barrier) {
        lock.lock()
        results.append(result)
        lock.unlock()
    }
}
```

### 2.5 高级场景

#### 场景1：DispatchWorkItem（可取消）

```swift
var workItem: DispatchWorkItem?

workItem = DispatchWorkItem {
    for i in 0..<1000000 {
        if workItem?.isCancelled ?? false {
            break
        }
        // 处理数据
    }
}

DispatchQueue.global().async(execute: workItem!)

// 取消任务
workItem?.cancel()
```

#### 场景2：DispatchBarrier（读写分离）

```swift
class Cache {
    private var storage: [String: Any] = [:]
    private let queue = DispatchQueue(label: "cache.concurrent", attributes: .concurrent)
    
    func read(key: String) -> Any? {
        // 读操作：可以并发
        return queue.sync {
            return storage[key]
        }
    }
    
    func write(key: String, value: Any) {
        // 写操作：barrier 确保独占
        queue.async(flags: .barrier) {
            self.storage[key] = value
        }
    }
}
```

#### 场景3：DispatchSource（事件源）

```swift
// 监听文件变化
let fd = open("/path/to/file", O_EVTONLY)
let source = DispatchSource.makeFileSystemObjectSource(fileDescriptor: fd,
                                                        eventMask: .write,
                                                        queue: .global())

source.setEventHandler {
    print("文件被修改")
}

source.resume()

// 定时器（不依赖 RunLoop）
let timer = DispatchSource.makeTimerSource(queue: .global())
timer.schedule(deadline: .now(), repeating: 1.0)
timer.setEventHandler {
    print("每秒执行，滑动列表不暂停")
}
timer.resume()
```

#### 场景4：信号量控制（同步异步）

```swift
// 等待多个异步任务完成
let semaphore = DispatchSemaphore(value: 0)
var result: Data?

DispatchQueue.global().async {
    result = fetchData()
    semaphore.signal()
}

semaphore.wait()
print("数据获取完成: \(result)")

// 控制最大并发数
let semaphore = DispatchSemaphore(value: 5)
for i in 0..<100 {
    DispatchQueue.global().async {
        semaphore.wait()
        performTask(i)
        semaphore.signal()
    }
}
```

#### 场景5：DispatchGroup 高级用法

```swift
let group = DispatchGroup()

// 进入 group
group.enter()
makeNetworkRequest { result in
    // 处理结果
    group.leave()
}

group.enter()
readFromDatabase { data in
    // 处理数据
    group.leave()
}

// 等待所有完成（阻塞当前线程）
group.wait()
print("两个任务都完成了")

// 或者异步通知
group.notify(queue: .main) {
    print("可以更新 UI 了")
}
```

## 三、调度策略对比表

### 3.1 队列类型对比

| 队列类型 | 创建方式 | 线程数 | 执行顺序 | 典型场景 |
|----------|----------|--------|----------|----------|
| **主队列** | `DispatchQueue.main` | 1（主线程） | 串行 | UI 更新 |
| **全局队列** | `DispatchQueue.global()` | 动态（线程池） | 并发 | 耗时任务 |
| **自定义串行** | `DispatchQueue(label:)` | 1 | 串行 | 资源保护、顺序执行 |
| **自定义并发** | `DispatchQueue(label:, attributes: .concurrent)` | 动态 | 并发 | 批量数据处理 |

### 3.2 QoS 优先级使用场景

| QoS | 优先级 | 使用场景 | 示例 |
|-----|--------|----------|------|
| **userInteractive** | 最高 | 动画、响应事件 | UI 刷新、用户手势 |
| **userInitiated** | 高 | 用户主动操作 | 打开文档、点击查询 |
| **default** | 中 | 默认优先级 | 普通业务逻辑 |
| **utility** | 低 | 可延迟的长时间任务 | 下载、数据导入 |
| **background** | 最低 | 后台维护任务 | 缓存清理、预加载 |

## 四、常见陷阱与最佳实践

### ❌ 常见错误

```swift
// 1. 在子线程更新 UI
DispatchQueue.global().async {
    self.label.text = "错误"  // 💥 可能崩溃
}

// 2. 主线程同步死锁
DispatchQueue.main.sync {
    // 💀 主线程死锁
}

// 3. 串行队列同步导致死锁
let queue = DispatchQueue(label: "serial")
queue.sync {
    queue.sync {
        // 💀 死锁
    }
}

// 4. 过度使用 sync
DispatchQueue.global().sync {
    // 阻塞当前线程，失去了异步的意义
}
```

### ✅ 最佳实践

```swift
// 1. 明确的线程边界
class ViewModel {
    private let queue = DispatchQueue(label: "viewmodel.serial")
    
    func fetchData() {
        queue.async {
            let data = self.loadData()
            DispatchQueue.main.async {
                self.updateUI(data)
            }
        }
    }
}

// 2. 使用 QoS 表达意图
DispatchQueue.global(qos: .userInitiated).async {
    // 表明这是用户发起的任务
}

// 3. 避免不必要的 sync
// ❌
let result = queue.sync { computeResult() }
// ✅ 如果需要等待结果，可以考虑回调
queue.async {
    let result = computeResult()
    DispatchQueue.main.async { callback(result) }
}

// 4. 使用 DispatchWorkItem 管理取消
let workItem = DispatchWorkItem {
    // 可取消的任务
}
queue.async(execute: workItem)
```

## 五、总结

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| UI 更新 | `DispatchQueue.main` | UIKit 非线程安全 |
| 网络请求 | `DispatchQueue.global(.userInitiated)` | 用户发起的操作 |
| 图片加载 | `DispatchQueue.global(.utility)` | 可延迟 |
| 文件读写 | 自定义串行队列 | 保证数据一致性 |
| 批量处理 | `DispatchQueue.concurrentPerform` | 并发效率高 |
| 资源保护 | 串行队列 + sync | 替代锁，性能更好 |
| 多任务等待 | `DispatchGroup` | 等待多个异步任务 |
| 并发限制 | `DispatchSemaphore` | 控制资源使用 |
| 定时任务 | `DispatchSourceTimer` | 不受 RunLoop 影响 |
| 可取消任务 | `DispatchWorkItem` | 支持取消操作 |
