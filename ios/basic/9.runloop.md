## iOS 线程管理与 RunLoop 详解

## 一、iOS 线程管理基础

### 1.1 线程的创建与成本

```objc
// 创建线程的三种方式
NSThread *thread = [[NSThread alloc] initWithTarget:self selector:@selector(task) object:nil];
[thread start];

[NSThread detachNewThreadSelector:@selector(task) toTarget:self withObject:nil];

// 更常用：GCD
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    // 子线程任务
});
```

**线程成本**：
| 项目 | 开销 |
|------|------|
| 内核数据结构 | 约 1KB |
| 栈空间 | 主线程 1MB，子线程 512KB |
| 创建耗时 | 约 90 微秒（相比 GCD 的 15 微秒）|

### 1.2 线程生命周期状态

```
新建(New) → 就绪(Runnable) → 运行(Running) → 阻塞(Blocked) → 死亡(Dead)
                ↑_________________↓
```

### 1.3 线程安全机制

```objc
// 1. NSLock
NSLock *lock = [[NSLock alloc] init];
[lock lock];
// 临界区
[lock unlock];

// 2. @synchronized
@synchronized(self) {
    // 临界区
}

// 3. dispatch_semaphore
dispatch_semaphore_t semaphore = dispatch_semaphore_create(1);
dispatch_semaphore_wait(semaphore, DISPATCH_TIME_FOREVER);
// 临界区
dispatch_semaphore_signal(semaphore);
```

## 二、RunLoop 详细介绍

### 2.1 什么是 RunLoop？

**RunLoop 是 iOS 事件驱动模型的核心**，本质是一个**运行循环**：

```objc
// RunLoop 的伪代码
function runLoop() {
    while (shouldKeepRunning) {
        // 1. 通知观察者：即将处理 Timer/Source
        notifyObservers(kCFRunLoopBeforeTimers);
        notifyObservers(kCFRunLoopBeforeSources);
        
        // 2. 处理 Timer 和 Source
        processTimers();
        processSources();
        
        // 3. 如果没有事件，进入休眠
        if (noMoreEvents()) {
            notifyObservers(kCFRunLoopBeforeWaiting);
            sleep(等待事件唤醒);
            notifyObservers(kCFRunLoopAfterWaiting);
        }
        
        // 4. 处理唤醒后的事件
        processWakeupEvents();
    }
}
```

### 2.2 RunLoop 的核心组件

```objc
// RunLoop 的结构
struct CFRunLoop {
    pthread_t _pthread;              // 绑定的线程
    CFMutableSetRef _commonModes;    // 通用模式集合
    CFMutableSetRef _commonModeItems;// 通用模式下的 items
    CFRunLoopModeRef _currentMode;   // 当前运行模式
    CFMutableSetRef _modes;          // 所有模式集合
};

// RunLoop Mode 的结构
struct CFRunLoopMode {
    CFStringRef _name;               // 模式名称
    CFMutableSetRef _sources0;       // Source0 集合
    CFMutableSetRef _sources1;       // Source1 集合
    CFMutableArrayRef _observers;    // 观察者数组
    CFMutableArrayRef _timers;       // 定时器数组
};
```

#### 2.2.1 CFRunLoopSource

| 类型 | 特点 | 示例 |
|------|------|------|
| **Source0** | 手动唤醒，需主动调用 `CFRunLoopSourceSignal` | `performSelector:`、触摸事件 |
| **Source1** | 基于 mach port，可自动唤醒 RunLoop | 系统事件、GCD 回调 |

#### 2.2.2 CFRunLoopTimer

```objc
// NSTimer 底层就是 CFRunLoopTimer
NSTimer *timer = [NSTimer scheduledTimerWithTimeInterval:1.0 
                                                  target:self 
                                                selector:@selector(task) 
                                                userInfo:nil 
                                                 repeats:YES];
// 等价于
CFRunLoopTimerRef cfTimer = CFRunLoopTimerCreate(NULL, CFAbsoluteTimeGetCurrent(), 1.0, 0, 0, timerCallback, NULL);
CFRunLoopAddTimer(CFRunLoopGetCurrent(), cfTimer, kCFRunLoopDefaultMode);
```

#### 2.2.3 CFRunLoopObserver

```objc
// 监听 RunLoop 状态变化
CFRunLoopObserverRef observer = CFRunLoopObserverCreateWithHandler(
    kCFAllocatorDefault,
    kCFRunLoopAllActivities,
    YES,
    0,
    ^(CFRunLoopObserverRef observer, CFRunLoopActivity activity) {
        switch (activity) {
            case kCFRunLoopEntry:        // 即将进入 RunLoop
                break;
            case kCFRunLoopBeforeTimers: // 即将处理 Timer
                break;
            case kCFRunLoopBeforeSources:// 即将处理 Source
                break;
            case kCFRunLoopBeforeWaiting:// 即将进入休眠
                break;
            case kCFRunLoopAfterWaiting: // 刚被唤醒
                break;
            case kCFRunLoopExit:         // 即将退出 RunLoop
                break;
        }
    });
CFRunLoopAddObserver(CFRunLoopGetCurrent(), observer, kCFRunLoopCommonModes);
```

### 2.3 RunLoop 的 Mode

| Mode 名称 | 说明 | 使用场景 |
|-----------|------|----------|
| `kCFRunLoopDefaultMode` | 默认模式 | 空闲状态处理事件 |
| `UITrackingRunLoopMode` | 追踪模式 | UIScrollView 滑动时 |
| `kCFRunLoopCommonModes` | 伪模式，标记集合 | 同时加入 Default 和 Tracking |

```objc
// 问题：滑动时定时器不工作
NSTimer *timer = [NSTimer scheduledTimerWithTimeInterval:1.0 
                                                  target:self 
                                                selector:@selector(update) 
                                                userInfo:nil 
                                                 repeats:YES];
// 滑动时 Timer 被暂停

// 解决：添加到 CommonModes
[[NSRunLoop currentRunLoop] addTimer:timer forMode:NSRunLoopCommonModes];
```

## 三、线程与 RunLoop 的关系

### 3.1 核心结论

> **RunLoop 与线程是一一对应的，每个线程都有且最多有一个 RunLoop。**

```objc
// 获取当前线程的 RunLoop
NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
// 主线程的 RunLoop 会自动创建并运行
// 子线程的 RunLoop 需要手动获取（懒加载创建）并手动运行
```

### 3.2 存储结构

```objc
// 全局字典：线程 -> RunLoop
static CFMutableDictionaryRef _runLoopMap;
static CFSpinLock_t _runLoopMapLock;

// 获取 RunLoop 的逻辑
CFRunLoopRef _CFRunLoopGet(pthread_t thread) {
    CFRunLoopRef runLoop = (CFRunLoopRef)CFDictionaryGetValue(_runLoopMap, thread);
    if (!runLoop) {
        // 创建新的 RunLoop
        runLoop = CFRunLoopCreate(thread);
        CFDictionarySetValue(_runLoopMap, thread, runLoop);
    }
    return runLoop;
}
```

### 3.3 主线程 vs 子线程的 RunLoop

| 特性 | 主线程 RunLoop | 子线程 RunLoop |
|------|---------------|----------------|
| **创建时机** | App 启动时自动创建 | 首次调用 `[NSRunLoop currentRunLoop]` 时懒加载 |
| **是否自动运行** | ✅ 是 | ❌ 否，需手动调用 `run` |
| **默认 Source** | 有（UI 事件、GCD、系统 Port） | 无 |
| **生命周期** | 随 App 结束 | 随线程结束 |

### 3.4 子线程使用 RunLoop

```objc
// 场景：子线程需要定时器或监听事件
dispatch_async(dispatch_get_global_queue(0, 0), ^{
    @autoreleasepool {
        NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
        
        // 添加定时器
        NSTimer *timer = [NSTimer timerWithTimeInterval:1.0 
                                                  target:self 
                                                selector:@selector(task) 
                                                userInfo:nil 
                                                 repeats:YES];
        [runLoop addTimer:timer forMode:NSDefaultRunLoopMode];
        
        // 添加 Port（防止 RunLoop 立即退出）
        [runLoop addPort:[NSMachPort port] forMode:NSDefaultRunLoopMode];
        
        // 启动 RunLoop
        [runLoop run];
    }
});
```

### 3.5 RunLoop 退出条件

```objc
// 子线程 RunLoop 退出的情况：
// 1. 没有 Timer/Source/Observer 时立即退出
// 2. 超时时间到
// 3. 手动调用 CFRunLoopStop()

[runLoop runUntilDate:[NSDate dateWithTimeIntervalSinceNow:10]]; // 10秒后自动退出

CFRunLoopStop(CFRunLoopGetCurrent()); // 手动停止
```

## 四、RunLoop 与 GCD 的关系

### 4.1 GCD 如何唤醒 RunLoop

```objc
// GCD 的回调通过 Source1 唤醒 RunLoop
dispatch_async(dispatch_get_main_queue(), ^{
    // 主队列的 block 会通过 mach port 发送消息
    // 唤醒主线程 RunLoop，然后在合适时机执行
});
```

**底层流程**：
```
dispatch_async(main_queue) 
    → _dispatch_send_wakeup() 
    → mach_msg() 发送消息到主线程的 mach port
    → RunLoop 被唤醒（如果正在休眠）
    → 处理 Source1 → 执行 block
```

### 4.2 为什么要保留 RunLoop？

| 有 RunLoop | 没有 RunLoop |
|-----------|-------------|
| 线程空闲时休眠，节省 CPU | 线程执行完任务立即退出 |
| 可响应事件（定时器、Port、Observer）| 无法接收异步回调 |
| 主线程的核心机制 | 一次性任务使用 |

## 五、常见面试题与陷阱

### Q1：performSelector:afterDelay 需要 RunLoop 吗？

```objc
[self performSelector:@selector(task) withObject:nil afterDelay:1.0];
// ✅ 需要 RunLoop（依赖 Timer）

dispatch_async(dispatch_get_global_queue(0, 0), ^{
    [self performSelector:@selector(task) withObject:nil afterDelay:1.0];
    // ❌ 子线程没有自动运行 RunLoop，不会执行！
    [[NSRunLoop currentRunLoop] run]; // 需要手动运行
});
```

### Q2：滚动时为什么 NSTimer 不工作？

```objc
// 原因：滚动时 RunLoop 切换到 UITrackingRunLoopMode
// Timer 默认添加到 NSDefaultRunLoopMode

// 解决方案 1：添加到 CommonModes
[[NSRunLoop currentRunLoop] addTimer:timer forMode:NSRunLoopCommonModes];

// 解决方案 2：使用 GCD Timer（不依赖 RunLoop）
dispatch_source_t timer = dispatch_source_create(DISPATCH_SOURCE_TYPE_TIMER, 0, 0, dispatch_get_main_queue());
dispatch_source_set_timer(timer, DISPATCH_TIME_NOW, 1.0 * NSEC_PER_SEC, 0);
dispatch_source_set_event_handler(timer, ^{
    // 始终在主线程执行，不受 RunLoop Mode 影响
});
dispatch_resume(timer);
```

### Q3：线程池中任务的 RunLoop 问题

```objc
// SDWebImage 等框架的做法
// 子线程下载图片，但需要在子线程处理数据，如何保持线程？
dispatch_async(backgroundQueue, ^{
    @autoreleasepool {
        // 创建并运行 RunLoop
        NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
        [runLoop addPort:[NSMachPort port] forMode:NSDefaultRunLoopMode];
        
        // 执行耗时操作
        UIImage *image = [self downloadImage];
        
        // 回调主线程
        dispatch_async(dispatch_get_main_queue(), ^{
            // 更新 UI
        });
        
        // 让 RunLoop 运行一段时间，等待可能的后续操作
        [runLoop runUntilDate:[NSDate dateWithTimeIntervalSinceNow:5]];
    }
});
```

## 三（续）。NSOperation 与 NSOperationQueue

### 3.1 NSOperation 的定位

```
┌─────────────────────────────────────────────────────────────┐
│                      并发编程 API 层次                        │
├─────────────────────────────────────────────────────────────┤
│  高层抽象  │  NSOperation / NSOperationQueue                │
│           │  - 面向对象，任务封装                            │
│           │  - 支持依赖、优先级、取消                        │
│           │  - KVO 友好                                      │
├───────────┼─────────────────────────────────────────────────┤
│  中层 API  │  GCD (Grand Central Dispatch)                   │
│           │  - C 语言，block 驱动                            │
│           │  - 队列管理，轻量高效                            │
├───────────┼─────────────────────────────────────────────────┤
│  底层 API  │  NSThread / pthread                             │
│           │  - 直接操作线程                                  │
│           │  - 需要手动管理生命周期                          │
└───────────┴─────────────────────────────────────────────────┘
```

### 3.2 NSOperation 的类型

```objc
// 1. NSInvocationOperation（iOS 11 后废弃）
NSInvocationOperation *op = [[NSInvocationOperation alloc] 
                              initWithTarget:self 
                              selector:@selector(task) 
                              object:nil];

// 2. NSBlockOperation（推荐）
NSBlockOperation *op = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"任务执行");
}];
// 可以追加多个 block
[op addExecutionBlock:^{
    NSLog(@"额外的任务");
}];

// 3. 自定义 NSOperation（复杂场景）
@interface MyOperation : NSOperation
@end

@implementation MyOperation
- (void)main {
    @autoreleasepool {
        if (self.isCancelled) return;
        // 执行任务
        [self doWork];
    }
}
@end
```

### 3.3 NSOperationQueue 的线程管理

```objc
// 创建队列
NSOperationQueue *queue = [[NSOperationQueue alloc] init];

// 设置最大并发数（直接管理线程数量）
queue.maxConcurrentOperationCount = 3;  // 最多同时 3 个任务

// 添加操作
[queue addOperation:operation];
[queue addOperationWithBlock:^{
    NSLog(@"block 操作");
}];

// 获取当前队列的线程信息
[queue addOperationWithBlock:^{
    NSLog(@"当前线程: %@", [NSThread currentThread]);
    // 不同的操作可能在不同的线程执行
    // NSOperationQueue 内部使用 GCD 的线程池
}];
```

### 3.4 NSOperation 的核心特性

#### 1. 依赖关系

```objc
NSOperation *op1 = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"步骤 1");
}];
NSOperation *op2 = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"步骤 2");
}];
NSOperation *op3 = [NSBlockOperation blockOperationWithBlock:^{
    NSLog(@"步骤 3");
}];

// op2 依赖 op1，op3 依赖 op2
[op2 addDependency:op1];
[op3 addDependency:op2];

[queue addOperations:@[op1, op2, op3] waitUntilFinished:NO];
// 执行顺序：op1 → op2 → op3

// 移除依赖
[op2 removeDependency:op1];
```

#### 2. 优先级

```objc
operation.queuePriority = NSOperationQueuePriorityHigh;    // 高
operation.queuePriority = NSOperationQueuePriorityNormal;  // 中（默认）
operation.queuePriority = NSOperationQueuePriorityLow;     // 低

// 注意：优先级只影响同一队列中未开始的任务
// 正在执行的任务优先级不生效
```

#### 3. 取消操作

```objc
// 单个取消
[operation cancel];

// 批量取消
[queue cancelAllOperations];

// 自定义操作中响应取消
- (void)main {
    @autoreleasepool {
        for (int i = 0; i < 1000000; i++) {
            if (self.isCancelled) {
                return;  // 立即退出
            }
            // 执行任务
        }
    }
}
```

#### 4. KVO 监听状态

```objc
// NSOperation 支持 KVO 的属性
// isFinished, isExecuting, isCancelled, isReady

[operation addObserver:self 
            forKeyPath:@"isFinished" 
               options:NSKeyValueObservingOptionNew 
               context:nil];

- (void)observeValueForKeyPath:(NSString *)keyPath 
                      ofObject:(id)object 
                        change:(NSDictionary *)change 
                       context:(void *)context {
    if ([keyPath isEqualToString:@"isFinished"]) {
        NSLog(@"操作完成");
    }
}
```

### 3.5 NSOperation 与 GCD 的对比

| 特性 | NSOperation | GCD |
|------|-------------|-----|
| **抽象层级** | 面向对象，高层 | C 语言，中层 |
| **取消操作** | ✅ 原生支持 | ❌ 需要手动实现 |
| **依赖管理** | ✅ `addDependency` | ❌ 需要手动 `dispatch_group` 或 `semaphore` |
| **最大并发数** | ✅ `maxConcurrentOperationCount` | ❌ 只能通过信号量模拟 |
| **优先级** | ✅ `queuePriority` | ✅ `dispatch_set_target_queue` + QOS |
| **KVO 监听** | ✅ 支持 | ❌ 不支持 |
| **性能** | 略低（OC 封装开销） | 更高（C 实现） |
| **队列挂起/恢复** | ✅ `suspended` | ✅ `suspend`/`resume` |

### 3.6 NSOperation 的最佳实践

#### 场景1：可取消的异步下载

```objc
@interface DownloadOperation : NSOperation
@property (nonatomic, copy) NSString *url;
@property (nonatomic, copy) void (^completion)(NSData *data);
@end

@implementation DownloadOperation {
    BOOL _executing;
    BOOL _finished;
}

- (void)start {
    if (self.isCancelled) {
        [self willChangeValueForKey:@"isFinished"];
        _finished = YES;
        [self didChangeValueForKey:@"isFinished"];
        return;
    }
    
    [self willChangeValueForKey:@"isExecuting"];
    _executing = YES;
    [self didChangeValueForKey:@"isExecuting"];
    
    // 异步下载
    NSURLSession *session = [NSURLSession sharedSession];
    NSURLSessionDataTask *task = [session dataTaskWithURL:[NSURL URLWithString:self.url] 
                                        completionHandler:^(NSData *data, NSURLResponse *response, NSError *error) {
        if (self.completion) self.completion(data);
        [self completeOperation];
    }];
    [task resume];
}

- (void)completeOperation {
    [self willChangeValueForKey:@"isExecuting"];
    [self willChangeValueForKey:@"isFinished"];
    _executing = NO;
    _finished = YES;
    [self didChangeValueForKey:@"isExecuting"];
    [self didChangeValueForKey:@"isFinished"];
}

@end
```

#### 场景2：限制并发数 + 依赖

```objc
// 限制同时下载数量为 2
NSOperationQueue *downloadQueue = [[NSOperationQueue alloc] init];
downloadQueue.maxConcurrentOperationCount = 2;

// 下载图片
NSBlockOperation *downloadOp = [NSBlockOperation blockOperationWithBlock:^{
    // 下载图片
}];

// 处理图片（依赖下载）
NSBlockOperation *processOp = [NSBlockOperation blockOperationWithBlock:^{
    // 处理图片
}];
[processOp addDependency:downloadOp];

// 上传结果（依赖处理）
NSBlockOperation *uploadOp = [NSBlockOperation blockOperationWithBlock:^{
    // 上传结果
}];
[uploadOp addDependency:processOp];

[downloadQueue addOperations:@[downloadOp, processOp, uploadOp] waitUntilFinished:NO];
```

#### 场景3：暂停/恢复队列

```objc
// 暂停队列（挂起）
queue.suspended = YES;  // 新任务不会开始，已开始的任务继续

// 恢复队列
queue.suspended = NO;

// 应用场景：滚动时暂停图片加载
- (void)scrollViewWillBeginDragging:(UIScrollView *)scrollView {
    self.imageQueue.suspended = YES;
}

- (void)scrollViewDidEndDragging:(UIScrollView *)scrollView willDecelerate:(BOOL)decelerate {
    if (!decelerate) {
        self.imageQueue.suspended = NO;
    }
}

- (void)scrollViewDidEndDecelerating:(UIScrollView *)scrollView {
    self.imageQueue.suspended = NO;
}
```

## 四、NSOperation 与 RunLoop 的关系

### 4.1 NSOperation 依赖 RunLoop 吗？

```objc
// NSOperation 本身不依赖 RunLoop
// 在子线程执行 NSOperation 时，任务执行完线程是否回收取决于队列

NSOperationQueue *queue = [[NSOperationQueue alloc] init];
[queue addOperationWithBlock:^{
    // 这个 block 执行在 GCD 的全局队列线程上
    // 执行完成后线程回到 GCD 线程池
    // 不涉及 RunLoop
}];

// 但如果 NSOperation 内部需要延时或定时器，需要 RunLoop
NSBlockOperation *op = [NSBlockOperation blockOperationWithBlock:^{
    // ❌ 这个不会执行（子线程没有 RunLoop）
    [self performSelector:@selector(delayedTask) withObject:nil afterDelay:1.0];
    
    // ✅ 需要手动运行 RunLoop
    [[NSRunLoop currentRunLoop] runUntilDate:[NSDate dateWithTimeIntervalSinceNow:1.0]];
}];
```

```
NSOperationQueue
       │
       │ 内部使用
       ↓
   GCD (libdispatch)
       │
       │ 管理
       ↓
   pthread 线程池
       │
```

### 4.3 常驻线程与 NSOperation

```objc
// 创建常驻线程执行 NSOperation
@interface ResidentThread : NSObject
@property (nonatomic, strong) NSThread *thread;
@property (nonatomic, strong) NSOperationQueue *queue;
@end

@implementation ResidentThread

- (instancetype)init {
    if (self = [super init]) {
        self.queue = [[NSOperationQueue alloc] init];
        self.thread = [[NSThread alloc] initWithTarget:self selector:@selector(threadEntry) object:nil];
        [self.thread start];
    }
    return self;
}

- (void)threadEntry {
    @autoreleasepool {
        // 给线程添加 RunLoop，使其常驻
        NSRunLoop *runLoop = [NSRunLoop currentRunLoop];
        [runLoop addPort:[NSMachPort port] forMode:NSDefaultRunLoopMode];
        
        // 将队列与当前线程关联（需要自定义实现）
        // 通常 GCD 已经处理好，这里仅示例
        
        [runLoop run];
    }
}

- (void)addOperation:(NSOperation *)op {
    [self.queue addOperation:op];
}

@end
```

## 五、完整的线程管理总结表

| API | 线程管理方式 | RunLoop 关系 | 适用场景 |
|-----|-------------|--------------|----------|
| **NSThread** | 手动创建/启动/退出 | 子线程需手动运行 | 需要直接控制线程 |
| **GCD** | 队列 + 线程池自动管理 | Source1 可唤醒 RunLoop | 高性能、轻量级任务 |
| **NSOperation** | 队列 + 并发数控制 + 依赖 | 不直接依赖 | 复杂任务管理、需要取消/依赖 |
| **RunLoop** | 线程事件循环机制 | 线程的附属物 | 需要事件响应、定时器 |

## 六、面试常见追问

### Q：NSOperation 和 GCD 如何选择？

| 需求 | 推荐 |
|------|------|
| 简单的异步任务 | GCD |
| 需要取消任务 | NSOperation |
| 需要任务依赖 | NSOperation |
| 需要限制最大并发数 | NSOperation |
| 需要 KVO 监听状态 | NSOperation |
| 极致性能 | GCD |
| 代码简洁性 | GCD |

### Q：NSOperationQueue 的 maxConcurrentOperationCount 和 GCD 的并发队列区别？

```objc
// GCD 并发队列：系统根据 CPU 核心数自动决定并发数
dispatch_queue_t queue = dispatch_queue_create("concurrent", DISPATCH_QUEUE_CONCURRENT);

// NSOperationQueue：可精确控制
queue.maxConcurrentOperationCount = 2;  // 同时最多 2 个任务
// 设置为 1：串行队列
// 设置为 NSOperationQueueDefaultMaxConcurrentOperationsCount：系统决定
```

### Q：如何实现一个可取消的 GCD 任务？

```objc
// GCD 原生不支持取消，需要手动实现
@interface CancellableTask : NSObject
@property (nonatomic, assign) BOOL cancelled;
- (void)cancel;
@end

@implementation CancellableTask
- (void)execute {
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        for (int i = 0; i < 1000000; i++) {
            if (self.cancelled) return;
            // 执行任务
        }
    });
}
- (void)cancel { self.cancelled = YES; }
@end

// 或者使用 dispatch_block_t 的取消功能（iOS 8+）
dispatch_block_t block = dispatch_block_create(0, ^{
    // 任务
});
dispatch_async(queue, block);
dispatch_block_cancel(block);  // 取消
```

---

这样补充后，NSOperation 的内容就完整了。现在线程管理的知识体系应该是：**NSThread（底层）→ GCD（中层）→ NSOperation（高层）→ RunLoop（线程附属机制）**，四个部分互相配合又各有侧重。

## Questions GCD Timer的调用

CD Timer 的核心 dispatch_source_t 是基于 kqueue 的内核级事件源，它的触发**不依赖 RunLoop，因此不会受 RunLoop Mode 切换的影响，在滑动列表时也能精准触发。

但是，任务的执行是否依赖 RunLoop，取决于传入的队列。如果目标是主队列，回调 Block 依然需要主线程的 RunLoop 来分发执行；如果是其他并发队列，GCD 会直接在线程池中的工作线程上执行，完全不依赖 RunLoop。”
