# Task.detached 完全解析

## 一、核心概念

`Task.detached` 创建一个**完全独立**的非结构化任务，它**不继承**父任务的任何上下文。

```swift
// 对比：结构化任务（继承上下文）
Task {
    // 继承父任务的优先级、Actor、取消状态等
}

// 对比：非结构化分离任务（完全不继承）
Task.detached {
    // 全新的独立任务，不继承任何上下文
}
```

## 二、关键区别

### 1. **Task（结构化任务）**

```swift
class ViewModel: ObservableObject {
    @MainActor
    func loadData() {
        Task {
            // ✅ 继承 MainActor 上下文
            // ✅ 继承父任务的优先级
            // ✅ 继承父任务的取消状态
            
            let data = await fetchData()
            self.data = data  // 自动在主线程（因为继承 MainActor）
        }
    }
}
```

### 2. **Task.detached（分离任务）**

```swift
class ViewModel: ObservableObject {
    @MainActor
    func loadData() {
        Task.detached {
            // ❌ 不继承 MainActor
            // ❌ 不继承父任务优先级
            // ❌ 不继承取消状态
            
            let data = await fetchData()
            
            // ⚠️ 需要在主线程更新 UI
            await MainActor.run {
                self.data = data
            }
        }
    }
}
```

## 三、完整对比表

| 特性 | `Task { }` | `Task.detached { }` |
|-----|-----------|---------------------|
| **继承 Actor 上下文** | ✅ 继承 | ❌ 不继承 |
| **继承优先级** | ✅ 继承 | ❌ 默认 `.medium` |
| **继承取消状态** | ✅ 父任务取消自动取消 | ❌ 独立生命周期 |
| **继承 TaskLocal 值** | ✅ 继承 | ❌ 不继承 |
| **父任务等待** | ✅ `async let` 会等待 | ❌ 不等待 |
| **使用场景** | 90% 的情况 | 特殊隔离需求 |

## 四、TaskLocal 的差异

```swift
// 定义任务局部值
enum TaskLocal {
    @TaskLocal static var requestID: String?
}

TaskLocal.$requestID.withValue("REQ-123") {
    Task {
        // ✅ 继承：可以访问 requestID = "REQ-123"
        print(TaskLocal.requestID ?? "nil")
    }
    
    Task.detached {
        // ❌ 不继承：requestID = nil
        print(TaskLocal.requestID ?? "nil")  // 输出 "nil"
    }
}
```

## 五、优先级继承差异

```swift
// 高优先级父任务
Task(priority: .userInitiated) {
    print("父任务优先级: high")
    
    Task {
        // ✅ 继承 .userInitiated
        print("子任务优先级: \(Task.currentPriority)")
    }
    
    Task.detached {
        // ❌ 默认 .medium，不继承
        print("分离任务优先级: \(Task.currentPriority)")
    }
}
```

## 六、取消传播差异

```swift
// 父任务
let parentTask = Task {
    let childTask = Task {
        // ✅ 父任务取消时，子任务自动取消
        try? await Task.sleep(nanoseconds: 1_000_000_000)
        print("子任务执行完成")
    }
    
    let detachedTask = Task.detached {
        // ❌ 父任务取消不影响分离任务
        try? await Task.sleep(nanoseconds: 2_000_000_000)
        print("分离任务执行完成")  // 即使父任务取消，仍会执行
    }
    
    try? await Task.sleep(nanoseconds: 500_000_000)
    parentTask.cancel()  // 取消父任务
}

// 输出：
// 分离任务执行完成（2秒后）
// 子任务被取消，不会输出
```

## 七、实际应用场景

### 1. 需要时使用 Task.detached

```swift
// 场景1：后台日志记录（不需要继承上下文）
func logEvent(_ event: String) {
    Task.detached(priority: .background) {
        // 独立的后台任务，不影响主任务
        await writeToLogFile(event)
    }
}

// 场景2：独立的后台清理
class CacheManager {
    func cleanExpiredCache() {
        Task.detached(priority: .low) {
            // 不继承当前上下文，低优先级执行
            await self.performCacheCleanup()
        }
    }
}

// 场景3：避免意外取消
func criticalOperation() async -> Result {
    Task.detached {
        // 即使父任务取消，这个操作也会继续执行
        // 适用于必须完成的操作（如保存数据）
        await saveCriticalData()
    }
}
```

### 2. 大多数情况不需要 Task.detached

```swift
// ❌ 不需要：普通异步操作
Task.detached {
    let data = await fetchData()
    await MainActor.run {
        self.updateUI(data)
    }
}

// ✅ 更好：使用普通 Task（继承 MainActor）
Task {
    let data = await fetchData()
    self.updateUI(data)  // 自动在主线程
}

// ❌ 不需要：网络请求
Task.detached {
    let result = await apiCall()
    // 处理结果...
}

// ✅ 更好：普通 Task 即可
Task {
    let result = await apiCall()
    // 处理结果...
}
```

## 九、性能考虑

```swift
// 普通 Task：可能复用现有线程
for i in 0..<1000 {
    Task {
        await doWork(i)
    }
    // 线程可能被复用，开销较小
}

// Task.detached：创建全新任务上下文
for i in 0..<1000 {
    Task.detached {
        await doWork(i)
    }
    // 每个都创建新上下文，开销更大
}
```

## 十、实际案例对比

### 场景：图片下载器

```swift
class ImageLoader {
    private var cache: [String: UIImage] = [:]
    private var activeTasks: [String: Task<Void, Never>] = [:]
    
    func loadImage(url: String, completion: @escaping (UIImage?) -> Void) {
        // ❌ 使用 detached 的问题
        let detachedTask = Task.detached {
            let image = await self.downloadImage(url: url)
            await MainActor.run {
                completion(image)
            }
        }
        
        // 问题：无法取消，无法追踪任务
        activeTasks[url] = detachedTask  // 类型不匹配
    }
    
    func loadImageBetter(url: String) async -> UIImage? {
        // ✅ 使用普通 Task
        let task = Task {
            return await downloadImage(url: url)
        }
        
        // 可以取消、可以等待
        activeTasks[url] = task as! Task<Void, Never>
        return await task.value
    }
}
```

## 十一、最佳实践

```swift
// 1. 默认使用 Task { }
Task {
    // 90% 的情况这里就够了
}

// 2. 只在明确需要隔离时使用 Task.detached
Task.detached(priority: .background) {
    // 独立的后台任务
}

// 3. 需要完全隔离的顶级任务
class AppDelegate {
    func applicationDidFinishLaunching() {
        Task.detached {
            // 应用启动时的独立任务
            await self.performStartupTasks()
        }
    }
}

// 4. 避免在 SwiftUI 视图中使用
struct MyView: View {
    var body: some View {
        Button("Load") {
            // ❌ 避免
            Task.detached {
                await loadData()
            }
            
            // ✅ 使用普通 Task
            Task {
                await loadData()
            }
        }
    }
}
```

## 十二、总结

| 方面 | Task | Task.detached |
|-----|------|---------------|
| **使用频率** | 90%+ | <10% |
| **上下文继承** | 继承 | 不继承 |
| **创建开销** | 小 | 大 |
| **取消传播** | 自动 | 需手动 |
| **优先级** | 继承 | 默认 `.medium` |
| **Actor 隔离** | 继承 | 不继承 |
| **主要用途** | 常规异步 | 完全独立任务 |

**核心原则**：
- **默认使用 `Task { }`**，它已经足够好
- **只在需要完全隔离时用 `Task.detached`**：
  - 不想继承父任务的取消
  - 需要完全独立的优先级
  - 顶级任务（如 App 启动）
  - 后台日志记录

**记忆口诀**：
- Task 继承上下文，日常开发最常用
- Detached 完全隔离，特殊场景才使用
- 取消优先级全独立，创建开销比较大
- 不是必须不要用，默认 Task 就足够
