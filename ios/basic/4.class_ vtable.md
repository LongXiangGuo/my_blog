# Class虚函数派发过程分析

## Swift 类的方法派发机制深度分析

## 场景一：虚函数派发（动态派发 / 多态）

### 核心概念

虚函数派发是指：**在运行时根据对象的实际类型来决定调用哪个方法实现**。这是实现多态的底层机制。

### SIL 层面的体现

从你的代码中，类 `C` 拥有一个 **vtable**（虚函数表）：

```swift
sil_vtable C {
  #C.value!getter: (C) -> () -> Int : @$s4demo1CC5valueSivg
  #C.value!setter: (C) -> (Int) -> () : @$s4demo1CC5valueSivs
  #C.value!modify: (C) -> () -> () : @$s4demo1CC5valueSivM
  #C.init!allocator: (C.Type) -> (Int) -> C : @$s4demo1CC5valueACSi_tcfC
  #C.deinit!deallocator: @$s4demo1CCfD
}
```

**vtable 结构**：

```
+-------------------+
| vtable for C      |
+-------------------+
| [0] getter        | ---> @$s4demo1CC5valueSivg
| [1] setter        | ---> @$s4demo1CC5valueSivs
| [2] modify        | ---> @$s4demo1CC5valueSivM
| [3] init          | ---> @$s4demo1CC5valueACSi_tcfC
| [4] deinit        | ---> @$s4demo1CCfD
+-------------------+
```

每个类实例（对象）的头部包含一个指向其 vtable 的指针（isa 指针）。

### 动态派发的调用过程

```swift
// Swift 源码
func callGetValue(_ obj: C) -> Int {
    return obj.value  // 调用 getter
}

// 生成的 SIL（伪代码）
sil @$s4demo11callGetValueySiAA1CCF : $@convention(thin) (C) -> Int {
bb0(%0 : $C):
    // 1. 从对象获取 vtable 指针
    %vtable = ref_vtable %0 : $C
    
    // 2. 从 vtable 中偏移获取 getter 函数指针
    %getter_fn = vtable_entry %vtable, #C.value!getter
    
    // 3. 通过函数指针进行间接调用
    %result = apply %getter_fn(%0) : $@convention(method) (C) -> Int
    
    return %result
}
```

**关键指令**（虽然不是标准 SIL，但表达逻辑）：
- `ref_vtable`：从对象引用获取 vtable 指针
- `vtable_entry`：从 vtable 中按索引取出函数指针
- `apply`：通过函数指针进行间接调用

### 为什么需要动态派发？

```swift
class Animal {
    func speak() { print("???") }
}

class Dog: Animal {
    override func speak() { print("Woof") }
}

class Cat: Animal {
    override func speak() { print("Meow") }
}

func makeSound(_ animal: Animal) {
    animal.speak()  // ❓ 编译时不知道是 Dog 还是 Cat
}

let d = Dog()
makeSound(d)  // 运行时才知道调用 Dog.speak()
```

## 场景二：方法重写

### 继承层次中的 vtable 演化

```swift
// Swift 源码
class Animal {
    func speak() { print("???") }
    func move() { print("Moving") }
}

class Dog: Animal {
    override func speak() { print("Woof") }
    func fetch() { print("Fetching") }
}
```

**vtable 结构变化**：

```
Animal vtable                  Dog vtable
+-------------------+          +-------------------+
| [0] speak         | -------> | [0] speak (override)
| [1] move          | -------> | [1] move (inherited)
+-------------------+          | [2] fetch (new)
                               +-------------------+
```

**关键点**：
1. 子类的 vtable **复制**父类的 vtable 布局
2. 重写的方法：子类 vtable 中的对应槽位指向**新的实现**
3. 继承的方法：槽位指向**父类的实现**
4. 新增的方法：在 vtable **末尾**添加新槽位

### SIL 层面的重写标记

虽然你的代码中没有重写，但 SIL 中会这样标记：

```swift
// 重写的方法会被标记 [override]
sil hidden [override] @$s4demo3Dog5speakyyF : $@convention(method) (@guaranteed Dog) -> () {
    // Dog.speak 的实现
}

// vtable 中的重写关系
sil_vtable Dog {
  #Animal.speak!override: (Dog) -> () -> () : @$s4demo3Dog5speakyyF
  #Animal.move!override: (Dog) -> () -> () : @$s4demo3Animal5moveyyF
  #Dog.fetch!1: (Dog) -> () -> () : @$s4demo3Dog5fetchyyF
}
```

### 内存布局示例

```swift
// 对象内存布局（简化）
class Animal {
    var name: String    // 8 字节（指针）
    func speak() {}     // vtable 索引 0
}

class Dog: Animal {
    var breed: String   // 额外 8 字节
    override func speak() {}  // vtable 索引 0 被覆盖
    func fetch() {}     // vtable 索引 1
}

// Dog 实例的内存布局
// [isa 指针] -> Dog.vtable
// [name] (8 bytes)
// [breed] (8 bytes)
```

调用 `dog.speak()` 时：
1. 通过 isa 指针找到 Dog 的 vtable
2. 访问 vtable[0]
3. 得到的是 Dog.speak 的实现（而不是 Animal.speak）

## 场景三：继承

### 初始化和 vtable 构建

从你的 SIL 代码中，我们可以看到类的初始化过程：

```swift
// C.__allocating_init(value:)
sil hidden [exact_self_class] @$s4demo1CC5valueACSi_tcfC : $@convention(method) (Int, @thick C.Type) -> @owned C {
bb0(%0 : $Int, %1 : $@thick C.Type):
    // 1. 分配堆内存（包含 isa 指针和引用计数）
    %2 = alloc_ref $C
    
    // 2. 设置 isa 指针指向 C 的 vtable
    //    （由 alloc_ref 隐式完成）
    
    // 3. 初始化字段（存储属性）
    %5 = end_init_let_ref %2
    %6 = ref_element_addr %5, #C.value
    store %0 to %6
    
    // 4. 返回初始化完成的对象
    return %5
}
```

### 子类继承的 SIL 表现

假设有一个子类 `SubC` 继承 `C`：

```swift
// Swift 源码
class SubC: C {
    var extra: String
    init(value: Int, extra: String) {
        self.extra = extra
        super.init(value: value)
    }
    override func someMethod() { ... }
}
```

**生成的 SIL 结构**：

```swift
// 1. 子类 vtable（包含父类方法 + 重写 + 新方法）
sil_vtable SubC {
  #C.value!getter: (C) -> () -> Int : @$s4demo3SubC5valueSivg  // 重写
  #C.value!setter: ... : @$s4demo3SubC5valueSivs
  #C.someMethod!override: ... : @$s4demo3SubC5someMethodyyF
  #SubC.extra!getter: ... : @$s4demo3SubC5extraSivg  // 新方法
}

// 2. 子类初始化器
sil hidden @$s4demo3SubC5value5extraACSi_SitcfC : $@convention(method) (Int, String, @thick SubC.Type) -> @owned SubC {
bb0(%0 : $Int, %1 : $String, %2 : $@thick SubC.Type):
    // 分配 SubC 对象（内存大小 = 父类大小 + 子类新增属性）
    %3 = alloc_ref $SubC
    
    // 初始化子类新增属性
    %4 = ref_element_addr %3, #SubC.extra
    store %1 to %4
    
    // 调用父类初始化器（链式初始化）
    %5 = upcast %3 : $SubC to $C  // 向上转型
    %6 = class_method %5 : $C, #C.init!allocator
    %7 = apply %6(%0, %5)  // super.init(value:)
    
    // 返回完成初始化的对象
    return %7
}
```

### 继承的关键 SIL 操作

| 操作 | SIL 指令 | 用途 |
|------|----------|------|
| **向上转型** | `upcast` | 将子类引用转为父类引用（类型擦除） |
| **向下转型** | `unchecked_ref_cast` / `conditional_cast` | 将父类引用转回子类 |
| **调用父类方法** | `super_method` / 通过 vtable 索引 | 在重写方法中调用父类实现 |
| **检查类型** | `checked_cast_br` | 运行时类型检查（`is` 操作符） |

### 方法派发的性能成本

```swift
// 静态派发（final / private）- 最快
final class FastClass {
    func fastMethod() {}  // 编译时确定地址
}

// 动态派发（非 final 类方法）- 稍慢
class NormalClass {
    func dynamicMethod() {}  // 通过 vtable 间接调用
}

// 消息派发（@objc 方法）- 最慢
class ObjcClass {
    @objc func objcMethod() {}  // 通过 Objective-C runtime
}
```

**性能对比**：
- 静态派发：直接调用（1 条指令）
- 动态派发（vtable）：2-3 条指令（取 vtable → 取函数指针 → 调用）
- 消息派发（objc_msgSend）：~10 条指令 + 缓存查找

## 完整派发决策树

```swift
方法调用派发方式决策流程：

是否标记为 final？
├── 是 → 静态派发
└── 否 → 是否在 private/fileprivate 作用域内？
    ├── 是 → 静态派发（编译器可证明无重写）
    └── 否 → 是否为类方法？
        ├── 是 → 是否标记为 @objc？
        │   ├── 是 → 消息派发（ObjC runtime）
        │   └── 否 → 动态派发（vtable）
        └── 否 → 结构体/枚举方法 → 静态派发
```

## 总结表格

| 场景 | 派发方式 | SIL 层体现 | 性能 | 灵活性 |
|------|----------|------------|------|--------|
| **虚函数派发（多态）** | 动态（vtable） | `ref_vtable` + 间接调用 | 中等 | 高 |
| **方法重写** | 动态（vtable 槽位覆盖） | vtable 中 `override` 标记 | 中等 | 高 |
| **继承** | 动态（vtable 扩展） | `upcast` + 父类初始化调用 | 中等 | 高 |
| **final 方法** | 静态 | 直接调用 | 高 | 无 |
| **结构体方法** | 静态 | 直接调用 | 高 | 无 |
| **@objc 方法** | 消息派发 | `objc_msgSend` | 低 | 最高 |

**关键洞察**：
1. Swift 的 vtable 机制与 C++ 类似，但增加了 ARC 的引用计数管理
2. 每个类实例的头部（isa）指向 vtable，实现运行时多态
3. final 和 private 可以帮助编译器将动态派发优化为静态派发
4. 结构体没有 vtable，所有方法调用都是静态的——这是 struct 比 class 快的另一个原因

你现在的 SIL 代码虽然简化，但已经包含了 vtable 的定义。如果要观察实际的动态派发过程，可以创建一个父类、子类并重写方法，然后用 `swiftc -emit-sil -O` 查看完整的 vtable 结构和派发指令。
