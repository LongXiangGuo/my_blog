### 1.2 预处理（Preprocessing）

预处理器对源代码进行文本级别的处理，主要完成以下工作：

- **头文件导入**：递归展开 `#import` 和 `#include` 指令，将被引用的头文件内容插入到当前文件中。
- **宏展开**：将 `#define` 定义的宏替换为对应的代码片段。
- **条件编译**：处理 `#if`、`#ifdef`、`#endif` 等指令，根据条件决定保留或删除代码块。
- **行号标记**：插入 `#line` 指令，供编译器在报错时能准确定位到原始源文件的行号。

经过预处理后，每个 `.m` 文件会生成一个**独立的、完整的编译单元**，包含所有展开的头文件内容和宏替换结果。

### 1.3 编译（Compilation）

编译是整个流程中最核心的阶段，由 **Clang 编译器前端** 和 **LLVM 编译器后端** 协作完成。

### 1.4 汇编（Assembly）

汇编器将编译阶段生成的汇编代码（如果编译器输出的是汇编文本）转换为机器可执行的二进制指令，生成**重定位目标文件（.o）**。该文件包含：

- **机器指令段**（Text Section）：程序的二进制指令
- **数据段**（Data Section）：初始化的全局变量和静态变量
- **符号表**（Symbol Table）：导出和引用的符号信息
- **重定位信息**（Relocation Information）：链接时需要修正的地址信息

在现代编译流程中，编译器通常直接生成 `.o` 文件，跳过独立的汇编步骤。

### 1.5 链接（Linking）

链接器（`ld`）将所有编译生成的 `.o` 目标文件以及依赖的系统库和静态库合并，生成最终的可执行文件（Mach-O）。

#### 链接的主要工作：

1. **符号解析（Symbol Resolution）**：
   - 收集所有 `.o` 文件中的**导出符号**（定义的函数、全局变量等）
   - 解析每个文件中引用的**外部符号**，在符号表中查找其定义位置

2. **地址重定位（Address Relocation）**：
   - 将各 `.o` 文件中函数调用指令的占位地址替换为最终的内存地址或偏移量
   - 处理跨模块的符号引用

3. **库搜索与加载**：
   - 静态库（`.a`）：将需要的对象文件提取并合并到可执行文件中
   - 动态库（`.dylib` / `.framework`）：记录依赖信息，运行时由 dyld 加载

#### Objective-C 链接的特殊处理：

- **`-ObjC` 标志**：强制链接器加载静态库中所有 Objective-C 类和 Category 的定义。如果不加此标志，链接器可能会为了优化体积而丢弃未被直接引用的 Category，导致运行时出现 `selector not recognized` 错误。

- **Category 方法查找**：OC 的 Category 方法是在运行时通过 `objc_msgSend` 动态查找的，链接器无法通过静态分析判断 Category 方法是否会被使用，因此需要 `-ObjC` 强制保留。

### 1.6 资源处理

在编译代码的同时，构建系统还会处理各种资源文件：

| 资源类型                   | 处理工具              | 输入                   | 输出                    | 说明                                      |
| -------------------------- | --------------------- | ---------------------- | ----------------------- | ----------------------------------------- |
| **Interface Builder 文件** | `ibtool`              | `.xib` / `.storyboard` | `.nib` / `.storyboardc` | 将可视化界面编译为运行时可读的二进制格式  |
| **资源目录**               | `actool`              | `.xcassets`            | `.car` (CoreUI Archive) | 将图片资源编译为优化的 Asset Catalog 文件 |
| **属性列表**               | `plutil` / `copytool` | `.plist`               | `.plist`                | 验证格式并复制到 Bundle                   |
| **其他资源**               | `copy`                | 图片、音频、字体等     | 原格式                  | 直接复制到 `.app` 包                      |

### 1.7 生成 .app 包

Xcode 的 `Copy Bundle Resources` 构建阶段会将编译好的可执行文件、所有处理后的资源文件、`Info.plist` 配置文件等复制到一个统一的目录结构中，形成 `.app` 应用包：

```
MyApp.app/
├── MyApp                    # 可执行文件（Mach-O）
├── Info.plist               # 应用配置文件
├── Assets.car               # 编译后的资源目录
├── Main.storyboardc/        # 编译后的 Storyboard
├── Base.lproj/              # 本地化资源
├── Frameworks/              # 嵌入的动态框架
└── PkgInfo                  # 包类型标识（可选）
```

### 1.8 归档（Archive）

在 Xcode 中通过 `Product > Archive` 执行归档操作时，构建系统会：

1. 使用 **Release 配置**（而非 Debug）重新编译，开启优化选项（如 `-O2`）
2. 生成 **`.xcarchive`** 包，其中包含：
   - **`.app` 包**：已编译的应用
   - **`.dSYM` 符号表文件**：用于后续解析崩溃日志，还原符号名称
   - **应用元信息**：版本号、构建号、签名信息等

### 1.9 代码签名与打包

归档后的应用需要经过代码签名才能分发和安装，这是苹果安全机制的核心环节。

#### 签名流程：

1. **准备签名材料**：
   - **开发者证书**：由 Apple Developer 签发，证明开发者身份
   - **私钥**：用于对应用进行数字签名
   - **描述文件（Provisioning Profile）**：包含允许安装的设备列表、应用 Entitlements（权限）等

2. **执行签名**：
   - `codesign` 工具使用私钥对 `.app` 包进行数字签名
   - 签名信息会被写入 `_CodeSignature` 目录
   - 描述文件被嵌入为 `embedded.mobileprovision`

3. **导出 .ipa**：
   - 签名后的 `.app` 包被压缩为 `.ipa`（iPhone Application Archive）
   - 企业分发时还会生成 `manifest.plist` 用于网页安装

│  │  编译产物    │───▶│  归档        │───▶│  代码签名    │           │
│  │  (.app)     │    │  (.xcarchive)│    │  (codesign)  │           │

### 2.3 SIL（Swift Intermediate Language）

SIL 是 Swift 编译器**独有的中间语言**，它比 LLVM IR 更高级，专门为 Swift 语言特性设计。SIL 保留了 Swift 的语义信息，使得编译器可以执行 Swift 特有的优化。

#### SIL 的关键特性：

1. **保留类型信息**：SIL 中保留了完整的类型信息，包括泛型参数、可选类型等。
2. **显式内存管理**：SIL 明确区分了强引用、弱引用和无主引用，将 ARC 操作显式化。
3. **方法派发表示**：SIL 使用不同的指令表示静态派发、虚表派发和消息派发。

#### SIL 优化示例：

```
// Swift 源代码
func process<T>(_ value: T) -> T {
    return value
}
let result = process(42)

// SIL 泛型特化（Generics Specialization）
// 编译器检测到 T 被实例化为 Int，生成专门针对 Int 的函数
// 避免运行时的类型装箱和类型检查
```

```
// Swift 源代码
class MyClass {
    func doSomething() { ... }
}
let obj = MyClass()
obj.doSomething()

// SIL 去虚拟化（Devirtualization）
// 编译器确定 obj 的实际类型是 MyClass，将 V-Table 调用转为静态调用
// 减少查表开销
```

```
// Swift 源代码
func add(_ a: Int, _ b: Int) -> Int { return a + b }
let result = add(3, 5)

// SIL 内联展开（Inlining）
// 编译器将 add 函数体直接嵌入到调用处
// 节省函数调用的栈帧开销
```

#### SIL 优化策略汇总：

| 优化类型       | 作用                        | 效果                |
| -------------- | --------------------------- | ------------------- |
| **泛型特化**   | 根据具体类型生成专用函数    | 避免运行时装箱/拆箱 |
| **去虚拟化**   | 将 V-Table 调用转为静态调用 | 减少查表开销        |
| **内联展开**   | 小函数体直接嵌入调用点      | 消除函数调用开销    |
| **ARC 优化**   | 减少多余的 retain/release   | 减少引用计数操作    |
| **死代码消除** | 移除未使用的代码            | 减小二进制体积      |
| **常量折叠**   | 编译期计算结果              | 减少运行时计算      |

### 2.5 链接阶段

Swift 工程的链接阶段与 OC 工程有以下关键区别：

#### Swift 标准库的链接方式

| 目标平台                 | 链接方式     | 原因                                                |
| ------------------------ | ------------ | --------------------------------------------------- |
| **iOS / tvOS / watchOS** | **静态链接** | 系统不保证预装 Swift 运行时，为兼容性将库打包进 App |
| **macOS**                | 动态链接     | 系统内置 Swift 运行时，App 直接依赖系统库           |

**静态链接的影响**：
- ✅ 优势：运行时版本与编译时一致，不受系统升级影响
- ❌ 劣势：二进制体积增加（通常增加 5-15 MB）

#### Swift 符号重整（Name Mangling）

Swift 编译器会对函数名进行重整（Mangling），将方法的完整类型信息编码到符号名中：

```
// Swift 源码
class MyClass {
    func doSomething(with value: Int) -> String { ... }
}

// 重整后的符号名（示例）
$s4MyApp7MyClassC11doSomething4withSSSi_tF

// 符号名编码了：模块名.类名.函数名.参数类型.返回值类型
```

**重整的作用**：
1. 支持函数重载（相同函数名不同参数类型产生不同符号）
2. 支持泛型特化（不同类型的泛型实例产生不同符号）
3. 实现命名空间隔离（不同模块的同名函数不冲突）

### 2.6 资源处理、归档、签名

Swift 工程在此阶段的流程与 OC 工程**完全一致**，因为 `.app` 包的结构和签名机制与语言无关：
- 使用 `ibtool` 编译 Storyboard/XIB
- 使用 `actool` 编译 Asset Catalog
- 使用 `codesign` 进行代码签名
- 通过 Archive 生成 `.xcarchive`
- 导出为 `.ipa` 分发包

### 4.3 步骤详解

#### 步骤 1：预处理与接口扫描

在正式编译开始前，构建系统需要让两种语言"互相认识"：

**OC 侧**：
- 预处理器处理 `#import` 和宏定义
- 扫描 OC 头文件，准备供 Swift 导入的接口列表

**Swift 侧**：
- Swift 编译器对 `.swift` 文件进行**浅解析**，只提取类、方法、属性的**声明**
- 重点关注标记了 `@objc` 的公开接口，暂不编译函数体内部实现

#### 步骤 2：生成"胶水"代码

这是混编流程中最特殊、最关键的一步，生成两种语言互相调用的桥梁：

**2.1 生成 `-Swift.h` 文件（自动）**

Swift 编译器收集所有标记了 `@objc` 的 Swift 类、协议、方法，自动生成一个 Objective-C 风格的头文件。

```
// Swift 源码
@objc class MySwiftClass: NSObject {
    @objc func doSomething() -> String { return "Hello" }
    @objc var name: String = ""
}

// 自动生成的 -Swift.h
@interface MySwiftClass : NSObject
- (NSString *)doSomething;
@property (nonatomic, copy) NSString *name;
@end
```

OC 代码只需 `#import "<ProjectName>-Swift.h"`，就能像调用普通 OC 类一样调用 Swift 类。

**2.2 读取桥接头文件（手动配置）**

开发者手动创建 `Bridging-Header.h`，在其中 `#import` 需要暴露给 Swift 的 OC 头文件。

```objc
// Bridging-Header.h
#import "MyOCTool.h"
#import <UIKit/UIKit.h>
#import "LegacyManager.h"
```

Swift 编译器读取该文件，将其中的 OC 接口映射为 Swift 语法，使 Swift 代码可以直接使用这些 OC 类。

> ⚠️ **注意**：这两个胶水文件必须在编译早期完成，因为 OC 代码编译时需要 `-Swift.h`，Swift 代码编译时需要桥接头文件。如果存在循环依赖（如 OC 头文件引用了 Swift 类，Swift 又引用了该 OC 类），编译会直接报错。

#### 步骤 3：并行编译

胶水代码生成后，两个编译器开始**并行**编译各自的源文件：

**OC 编译流水线（Clang）**：
```
.m 文件 → 预处理 → 词法/语法分析 → AST → LLVM IR → .o 目标文件
```
此时，OC 代码中通过 `-Swift.h` 引用的 Swift 类被视为**外部符号**，留到链接时解析。

**Swift 编译流水线（swiftc）**：
```
.swift 文件 → AST → SIL 生成 → SIL 优化 → LLVM IR → .o 目标文件
```
Swift 代码中调用的 OC 方法（通过桥接头文件映射）也被记录为**外部符号**。

> 💡 **性能提示**：Xcode 会利用多核 CPU 并行编译多个 `.m` 和 `.swift` 文件。混编项目首次完整编译通常比纯语言项目慢，因为需要启动两套编译前端（Clang + swiftc）并生成额外的胶水代码。

#### 步骤 4：链接

所有 `.o` 文件生成后，链接器开始工作：

1. **符号解析**：
   - 收集所有 `.o` 文件和依赖库中的符号表
   - OC 调用的 Swift 类方法 → 在 Swift `.o` 中找到实现
   - Swift 调用的 OC 类方法 → 在 OC `.o` 或系统库中找到实现

2. **Swift 符号重整**：Swift 编译器对符号进行了重整，链接器需要正确解析 Mangled 符号

3. **地址重定位**：将函数调用指令的占位地址替换为实际内存地址

4. **生成可执行文件**：输出完整的 Mach-O 可执行文件

> ⚠️ **常见错误**：
> - `Undefined symbol: _OBJC_CLASS_$_MySwiftClass`：Swift 类未标记 `@objc`
> - `Undefined symbol: _OBJC_CLASS_$_MyOCClass`：OC 类未正确导入桥接头文件

#### 步骤 5：打包与签名

与纯语言项目完全一致，不再赘述。

### 5.3 混编优化策略

| 优化方向                      | 具体建议                                      | 效果                                   |
| ----------------------------- | --------------------------------------------- | -------------------------------------- |

## Q: 什么是去虚拟化

“去虚拟化”（Devirtualization）是Swift编译器在**SIL（Swift Intermediate Language，Swift中间语言）** 优化阶段所做的一项关键优化，它属于Swift编译器优化管线中的一环。

简单来说，它的核心原理是**在编译期，将那些在运行时通过查“虚表”（V-Table）来确定调用地址的动态派发，转化为编译期就已经确定的静态直接调用**。

### ⚙️ 原理与示例：从查表到直调

*   **动态派发 (Table Dispatch) 的过程**：当调用类的一个实例方法时，编译器无法确定这个实例的具体类型。它会通过对象找到其所属类的**虚函数表（V-Table）**，再根据方法在表中的**固定偏移量**找到对应的函数地址，然后调用。这涉及“查表-寻址-跳转”三个步骤，有一定开销，且因为运行时才知道调用哪个方法，编译器无法做内联等优化。

*   **去虚拟化 (Devirtualization) 的原理**：编译器在SIL优化阶段进行分析。当它发现某个变量的实际类型在上下文中是确定的（例如，已知它是`Dog`类的一个实例，而不是它的父类`Animal`），就可以确定要调用的具体是哪个方法实现。于是，编译器将原本的“查表调用”替换为对那个具体函数实现的“**静态直接调用**”。

*   **代码示例**：
    ```swift
    // 父类
    class Animal {
        func makeSound() { print("Some animal sound") }
    }
    // 子类
    class Dog: Animal {
        override func makeSound() { print("Woof!") }
    }

    // 场景1: 动态派发 (需要查表)
    let animal: Animal = Dog()
    animal.makeSound() // 这里无法确定具体类型，编译器会使用 V-Table 派发

    // 场景2: 去虚拟化 (直接调用)
    let dog = Dog()    // dog 的类型明确是 Dog
    dog.makeSound()    // 编译器优化后，会直接调用 Dog 类的 makeSound 实现，无需查表
    ```

### 🗂️ Swift中的“表”：V-Table、Witness Table 和 Message Dispatch

Swift为了在性能和灵活性之间取得平衡，主要使用了以下几种派发机制，你可以将它们理解为不同的“函数名册”：

| 派发机制                                         | 别名/相关表                      | 主要使用场景                                                                                                                                                                                           | 特点与作用                                                                                                                                    |
| :----------------------------------------------- | :------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------- |
| **静态派发 (Static Dispatch)**                   | 直接调用 (Direct Call)           | - **值类型**：结构体（`struct`）和枚举（`enum`）的所有方法<br>- **`final` 关键字**修饰的类或类方法<br>- **`private` 关键字**修饰的方法（在声明作用域内）<br>- 类在**扩展（`extension`）** 中定义的方法 | **最快**。编译时直接确定函数地址，无运行时开销，支持编译器内联优化。应优先使用。                                                              |
| **虚表派发 (Table Dispatch)**                    | **V-Table** (Virtual Table)      | **普通的类（`class`）实例方法**（未使用`final`、`private`或`static`修饰）                                                                                                                              | **较快**。通过类的虚函数表实现多态。每个类维护一个函数指针数组，子类重写方法会覆盖对应条目。这是纯Swift环境下实现继承和多态的主要方式。       |
| **协议表派发 (Protocol Witness Table Dispatch)** | **PWT** (Protocol Witness Table) | 当通过**协议类型**（如`let a: SomeProtocol`）调用协议中定义的**必需方法**时                                                                                                                            | 实现**面向协议编程**的动态派发。每个遵守协议的类型都会生成一个“协议见证表”，用于在运行时查找该类型对协议方法的具体实现。                      |
| **消息派发 (Message Dispatch)**                  | `objc_msgSend`                   | - 类或方法标记了 **`@objc dynamic`**<br>- 继承自 `NSObject` 的类，且方法使用了 `@objc`                                                                                                                 | **最灵活，但最慢**。使用Objective-C的运行时消息发送机制，支持方法交换（Swizzling）等动态特性。仅在需要与Objective-C交互或使用动态特性时使用。 |

### 💡 总结

去虚拟化是Swift编译器在SIL层面做的一个“智能优化”，通过把运行时的查表调用提前到编译期，换取性能提升。理解Swift不同的派发表（V-Table、PWT等），有助于我们写出更高效、更符合Swift设计哲学的代码。通常，优先使用静态派发（如`struct`、`final`）能获得最佳性能，而协议表派发则为Swift的面向协议编程提供了动态能力。

## Q: 什么是双向注册

这个问题问到了Swift与Objective-C互操作的核心机制。你提到的这几个概念——**符号名称（Symbol Name）**、**普通符号（Undecorated/Unmangled）**、**Mangled（重整/修饰）符号**以及**双向注册（Bidirectional Registration）**——环环相扣，共同构成了Swift/ObjC动态调用的基础。

为了让你彻底理解，我们从底层到应用层拆解：

### 1. 符号名称（Symbol Name）是什么？
在编译后的二进制文件（Mach-O）中，**符号（Symbol）** 是函数、全局变量、类名的“代号”。链接器（Linker）和运行时（Runtime）靠符号来找到对应的代码地址。

- **在 ObjC 中**：符号通常是**类名 + 方法名 + 参数编码**（如 `-[Person setName:]`）。
- **在 Swift 中**：符号分为两种形态（即你提到的“普通”和“Mangled”）。

---

### 2. 普通符号 vs. Mangled（重整）符号

#### A. 普通符号（Undecorated / Unmangled Symbol）
- **定义**：**不带类型信息的**、人类可读的符号名称。
- **典型场景**：**纯 Objective-C** 代码，或者 Swift 类被标记为 `@objc` 后暴露给 ObjC 时的**别名（Alias）**。
- **示例**：
  ```objectivec
  // ObjC 方法
  - (void)updateUI;
  // 符号名（在 Mach-O 中）通常为：
  _updateUI
  // 或者带有类名：
  -[ViewController updateUI]
  ```
- **特点**：仅包含方法名和类名，**不包含**参数类型、返回值类型。这意味着如果两个方法同名但参数不同（重载），在 ObjC 中会冲突（因为符号无法区分）。

#### B. Mangled（重整）符号（Mangled Symbol）
- **定义**：**包含完整类型信息**的、经过编译器“编码”的符号。
- **典型场景**：**纯 Swift 代码**（不暴露给 ObjC 的函数、泛型、重载函数）。
- **示例**（Swift 编译后的实际符号）：
  ```swift
  func process(value: Int) -> String { ... }
  // 其 Mangling 后的符号可能看起来像：
  $s4MyApp7process5valueSiSS_tF
  ```
  解读：`$s`（全局标识） → `4MyApp`（模块名） → `7process`（函数名） → `5value`（参数标签） → `Si`（Int 类型） → `SS`（String 类型） → `tF`（函数结束标记）。
- **特点**：
  1. **全球唯一**：因为包含了参数类型、返回值类型、模块名，**支持函数重载**（`process(Int)` 和 `process(String)` 的 Mangled 名不同）。
  2. **无法被 ObjC 运行时直接识别**：ObjC 是基于“字符串名称”查找方法的，它看不懂这种带 `$s` 和类型编码的符号。

---

### 3. “双向注册”解决了什么问题？

由于 Swift 代码（Mangled 符号）和 ObjC 代码（普通符号）在运行时“语言不通”，为了让它们能互相调用，编译器引入了 **双向注册（Bidirectional Registration）** 机制。它实际上包含两个方向的映射：

#### 方向一：Swift → ObjC（暴露给 Objective-C）
当你在 Swift 类/方法前加 `@objc` 时，编译器会做两件事：
1. **保留 Mangled 符号**（Swift 内部调用依然用它，支持重载和类型安全）。
2. **生成一个“普通符号”别名**，并将其**注册到 ObjC 运行时**（即 `objc_class` 和 `objc_method` 列表中）。
   - 这个别名默认是 `类名.方法名`（如 `Person.updateUI`），你也可以通过 `@objc(updateUI:)` 自定义。
   - **关键点**：此时 **ObjC 运行时** 的符号表中多了一条记录，指向这个 Swift 函数的实现地址。

#### 方向二：ObjC → Swift（导入到 Swift）
当你在 Swift 中继承 `NSObject` 或调用 ObjC API 时：
1. 编译器会读取 ObjC 头文件（`.h`）。
2. 将这些 **ObjC 的普通符号**映射为 Swift 可识别的 **Mangled 符号**（或者直接生成 Swift 的调用桩 Stub）。
3. **关键点**：Swift 编译器在编译期会在自己的“模块表”中注册这些 ObjC 类和方法，让 Swift 代码能像调用原生 Swift 函数一样调用它们（但实际上底层走的依然是 `objc_msgSend`）。

---

### 4. 为什么需要双向注册？（深层机制图解）

| 场景                     | 编译时的符号形态                                                       | 运行时的查找方式                             | 注册方                                                       |
| :----------------------- | :--------------------------------------------------------------------- | :------------------------------------------- | :----------------------------------------------------------- |
| **纯 Swift 调用 Swift**  | Mangled（含类型）                                                      | 直接地址调用（静态）或 Swift V-Table（虚表） | 无需注册，由 Swift 内部维护                                  |
| **Swift 调用 ObjC 方法** | Swift 侧编译为调用 `objc_msgSend`，入参为 ObjC 的 Selector（普通符号） | ObjC 运行时查类名/方法名（普通符号）         | **ObjC 运行时**（系统已注册）                                |
| **ObjC 调用 Swift 方法** | 编译时生成 ObjC 兼容的桩代码（Thunk），暴露**普通符号**                | ObjC 运行时查到该普通符号，跳转至 Swift 实现 | **Swift 编译器主动注册**到 ObjC 运行时（`@objc` 触发的注册） |

---

### 5. 形象类比（帮助你记忆）

- **普通符号**：相当于一个只有**姓名**的人（`张三`）。如果有两个同名同姓的（重载），就分不清了。ObjC 的世界就是靠名字喊人。
- **Mangled 符号**：相当于**身份证号**（`110101...`）。包含出生地、日期、序号，全球唯一。Swift 的世界里靠身份证号精确找人。
- **双向注册**：就像是**外交部**。
  - **Swift → ObjC（@objc）**：给 Swift 函数发一本“外国护照”（普通符号别名），并通知 ObjC 海关（运行时）这个外国人可以入境。
  - **ObjC → Swift（导入）**：把 ObjC 的海关记录（头文件）翻译成 Swift 语言，告诉 Swift 市民（编译器）怎么去 ObjC 地界找人办事。

### 6. Mangled作用

> *“Mangled 符号是 Swift 为了支持函数重载、泛型和命名空间而引入的包含完整类型信息的唯一标识符；而普通符号是不含类型信息的简名，主要用于 Objective-C 动态运行时。两者不互通，所以 Swift 通过 `@objc` 和 `@objcMembers` 机制，在编译期生成一个 Mangled 符号到普通符号的映射表，并将该映射注册到 ObjC 运行时的类和方法缓存中，这个过程叫做双向注册。这使得 ObjC 运行时可以通过字符串名称查找到 Swift 函数实现，反之 Swift 也能通过桥接桩调用 ObjC 方法。”*

**额外进阶点**：如果你关注性能，Swift 内部尽量使用 Mangled 符号做直接调用（去虚拟化），只有在跨语言边界（`@objc`）时，才会退化为 `objc_msgSend`（消息派发），这是其性能优于纯 ObjC 的原因之一。
