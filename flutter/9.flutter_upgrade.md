## 升级前准备

```
========================================================================================
                                 FLUTTER 升级前准备任务清单
========================================================================================
 📂 一、代码与环境隔离（基线备份）
 └── 🧱 1. 检出隔离分支：禁止在开发分支操作，创建 `{team}/technical/flutter-{version}`分支
 └── 📝 2. 生成环境快照：运行 `flutter doctor -v > before_upgrade.log` 保存升级前基准
 └── 🧹 3. 彻底清理缓存：执行 `flutter clean`，防止旧版本 Android/iOS/鸿蒙中间件干扰

 ⚙️ 二、三端原生工具链对齐（环境预检）
 ├── 🍏 1. iOS 侧 (Xcode & CocoaPods)
 │   ├── 核对目标 Flutter 版本所需的最低 Xcode 版本限制
 │   └── 更新本地 Pod 环境：执行 `sudo gem install cocoapods` 及 `pod repo update`
 ├── 🤖 2. Android 侧 (Gradle & JDK)
 │   ├── 确认目标版本需要的 JDK（新版通常强制要求 JDK 17 及以上）
 │   └── 提前通过 Android Studio 下载好匹配的最新 SDK Platform 工具包
 └── ⛰️ 3. 鸿蒙侧 (DevEco & HarmonyOS NEXT) 🆕
     ├── 确认 Flutter 鸿蒙分支对应的 HarmonyOS NEXT SDK 目标基线（如 API 12/13/14） [1]
     ├── 通过 DevEco Studio 提前下载匹配的 Native / ArkTS 编译工具链
     └── 检查环境变量（如 `HD_SDK_PATH`），确保终端可以正常调用 `hdc`等命令行

 📦 三、依赖项与插件扫描（风险评估）
 ├── 📊 1. 运行过时检测：执行 `flutter pub outdated --mode=null-safety` 摸底三方包现状
 └── 🔍 2. 核心插件专项核对清单
     ├── 基础底座：`provider` / `analyze` / `flutter_bloc`/`build_runner` 确认是否需要跟进大版本
     ├── 国内特化 (MyApp China)：高德/百度地图、微信/支付宝 SDK、火山引擎等是否发布兼容包
     └── 鸿蒙特化 🆕：确认核心业务插件的 ArkTS 侧 `MethodChannel` 桥接层是否已适配

 🚨 四、熔断与回退机制（风险红线）
 ├── ⏱️ 1. 设定时间盒 (Timebox)：例如分配 3 个工作日锁定，若 3 天后仍有编译阻断则立即熔断
 └── 🔄 2. 准备一键回退预案
     ├── 代码回退：直接废弃当前隔离分支，无痛切回业务主干
     └── 引擎回退：保留旧版 Flutter SDK 根目录，随时可通过修改 PATH 或使用 FVM 一键恢复
========================================================================================
```

## 升级步骤

### 项目优化思路

- 由“单线升级”改为“分阶段升级 + 并行”模式,优先保证其中一个版本android构建成功(android依赖链环境相比iOS要多,变化也快),然后同步并行构建iOS。
- 所有步骤都定义输入(编译构建工具链版本,插件版本以及修改内容)、输出(构建版本链接)、通过标准，管理各构建版本改动的差异,方便后需逐步升级到稳定版本和bug追踪。
- 插件处理遵循“先 override 保命、后回归官方版本、最后清理临时补丁”的三段式策略。

### 阶段化执行（推荐）

#### Phase 0: 版本立项与影响评估（0.5~1 天）
1. 锁定候选 Flutter 版本（建议 2 个：目标版 + 回退保底版）。
2. 输出影响清单：
    - 平台工具链：Xcode/JDK/Gradle/DevEco。
    - 核心插件：地图、webview、蓝牙、音视频、countly、火山引擎、支付与登录能力。
    - 核心模块：carconnection、remote3d、rke、bluetooth、map、webview。
3. 产出《升级 RFC》：范围、风险、回退策略、里程碑。

通过标准：owner、时间盒、回退路径全部明确。

#### Phase 1: 依赖冻结与可编译版本（0~3 天）
1. 生成依赖图（`flutter pub deps --style=compact`）与过时报告（`flutter pub outdated`）。
2. 分层锁定依赖：
    - L1 Flutter/Dart 自带基础库。
    - L2 基础插件（IO/网络/图片/加密/socket/音视频/蓝牙）。
    - L3 业务插件（video_picker/webview/countly 等）。
    - L4 公司内部插件。
3. 对阻塞插件采用临时 `dependency_overrides` 或缓存补丁，目标仅是“本地 编译通过”。
4. 若 3 天内无法达成可编译版本，立即触发熔断：切回保底版本或缩小升级范围。

通过标准：至少Android/iOS代表变体可编译,主流程可启动。

#### Phase 2: 去临时补丁与主链路打通（第 4~7 天）
1. 逐个移除临时 override/缓存补丁，替换为可审计的正式版本。
2. 阅读 changelog 和 breaking changes，优先处理 API 重构与生命周期变更。
3. 提交 Draft PR，接入 POC pipeline，验证以下主链路：
    - 登录/启动/首页
    - 车连接与远程控制
    - 地图,WebView,扫码
    - 具有复杂ui特效的功能,如trip图表,充电
    - Push/通知/埋点
    - Flutter生命周期

通过标准：Draft PR 可稳定构建，主链路冒烟通过。

#### Phase 3: 分模块并行修复（第 8~14 天）
1. 按错误类型建看板并行处理：
    - 编译错误
    - 单元测试失败
    - analysis/lint
    - 平台专项错误（iOS/Android/鸿蒙）
2. 每个模块设单点 owner，按 feature 批次合入修复。
3. 每日同步风险：新增 blocker 必须标注“是否影响合并窗口”。

通过标准：目标模块 pipeline 全绿，阻塞项清零或有明确豁免说明。

#### Phase 4: 回归、合并与发布准备（第 15~17 天）
1. 出具升级差异清单：工具链、插件版本、breaking changes、受影响 feature。
2. 执行多端回归（含 Brand/Region/Platform 变体关键集），重点关注鸿蒙桥接与原生插件稳定性。
3. 完成 PR review 合并与发布前 Go/No-Go 评审。

通过标准：回归通过 + 评审通过 + 回退预案可执行。

## 进度管控

建议采用“里程碑”而不是纯日期描述：

- M1（D3）：本地编译通过（至少Android 1 个代表变体）。
- M2（D7）：Draft PR + POC主链路冒烟通过。
- M3（D14）：pipeline 全绿（编译/UT/analysis/平台专项）。
- M4（D17）：回归通过 + Go/No-Go 通过 + 可发布。

建议每晚同步以下指标：

- 编译通过率（按平台、按变体,按POC构建,pipeline检测项目）
- UT 通过率
- analysis 告警数
- blocker 数量

## 问题追踪

建议建立统一 issue 台账（Confluence/Jira 都可），字段最少包含：

- 问题分类（编译/UT/analysis/运行时/平台特有）
- 影响范围（模块 + 平台 + 变体）
- 严重级别（S1/S2/S3）
- 临时规避方案
- 根因与永久修复方案
- owner 与截止时间

重点追踪来源：

- Flutter 官方 issue 与 release notes
- 关键三方插件 changelog
- 宝马内部插件仓库升级记录

## PR review合并

合并前必须附带升级清单（建议模板化）：

- 工具链变更（Flutter/Dart/Xcode/JDK/Gradle/DevEco）
- 插件变更（版本前后对比 + breaking changes）
- 临时补丁清理情况（是否已全部移除）
- 影响模块与风险说明（carconnection/remote3d/rke/bluetooth/map/webview 等）
- 当前运行环境快照（`flutter doctor -v`）
- 回退方案（分支回退 + SDK 回退）

## 回归测试

建议采用“两层回归”：

- L1 快速回归：主链路 + 高风险模块（每天）
- L2 完整回归：发版前全量执行（含多变体）

必须覆盖：

- 启动性能与首屏稳定性
- 登录与账号链路
- 车连接/远程控制/蓝牙
- 地图/WebView/支付/Push/埋点
- 崩溃率与关键性能指标对比（升级前后）

## 总结

回顾建议固定输出 3 份资产：

- 升级复盘（问题、根因、收益、成本）
- 可复用脚本（依赖扫描、环境校验、回归命令集）
- 下一次升级建议（是否继续跟进次新版本）
